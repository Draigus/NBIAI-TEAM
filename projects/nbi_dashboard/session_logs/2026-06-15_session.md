**Evidence:** Current `shell-guard.js` has `resolveCommand()` looping through wrapper layers and the bundled test suite passes: `node .claude/harness/tests/test-shell-guard.js` returned 99 passed, 0 failed. Direct probes confirmed all five required round 5 commands block: `sudo env FOO=bar bash -lc ... cp governed`, `env FOO=bar sudo bash -lc ... cp governed`, `sudo command git -C dir restore governed`, `env FOO=bar command git restore governed`, and `sudo env -i git restore governed`. Additional three-deep probes also blocked: `sudo env FOO=bar command bash -lc ...`, `env FOO=bar command sudo bash -lc ...`, `command sudo env FOO=bar git -C dir restore ...`, and `xargs -I{} sudo env FOO=bar bash -lc ...`. Controls passed: `sudo cat .claude/harness/lib/write-guard.js` allowed and `env git status` allowed.

**Outcome:** Not a clean PASS. `env -S` remains a bypass. Confirmed allowed governed writes:
- `env -S "bash -lc" "cp /tmp/x .claude/harness/lib/evil.js"`
- `sudo env -S "bash -lc" "cp /tmp/x .claude/harness/lib/evil.js"`
- `env -S "sudo bash -lc" "cp /tmp/x .claude/harness/lib/evil.js"`
- `command env -S "sudo bash -lc" "cp /tmp/x .claude/harness/config/x.json"`

**Root cause:** `resolveCommand()` tokenises with `segment.trim().split(/\s+/)` and treats `-S` as an argument-taking flag to skip one token. That is insufficient for `env -S`, whose quoted split string is itself the command to execute. The resolver skips `"bash` then sees `-lc"` as another option-like token and ultimately resolves to a quoted inner argument rather than `bash`, so shell-exec analysis never runs. `env -S "git status"` remains allowed, so the false-positive control for benign `env -S` status is sane.
