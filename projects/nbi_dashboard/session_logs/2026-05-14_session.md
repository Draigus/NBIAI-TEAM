## Entry 1: Sidebar scroll root cause found

**Investigation:** Systematic debugging of sidebar scroll failure. Traced the CSS chain: `.shell` → `.sidebar` → `.sidebar__nav`. The CSS layout was correct on paper (`flex: 1; overflow-y: auto`), but Playwright diagnostics revealed `--vh-full` was resolving to empty string.

**Root cause:** Line 30 of `nbi_project_dashboard.html` contained a self-referential CSS variable:
```css
@supports (height: var(--vh-full)) { :root { --vh-full: var(--vh-full); } }
```
Per CSS spec, a custom property referencing itself is "guaranteed-invalid." Since all modern browsers support `var()`, the `@supports` condition is always true, so this rule fires and destroys the valid `--vh-full: 100vh` from line 29.

**Impact:** `calc(var(--vh-full) - var(--header-h))` on `.shell` resolved to `height: auto` — no height constraint on the entire layout shell. Without a constrained shell height, the sidebar grew to fit its content and never needed to scroll. `fixScrollHeights()` partially compensated via JS-calculated pixel heights, but the underlying CSS chain was broken.

**Fix:** Deleted line 30 (the `@supports` rule). Verified via Playwright:
- Before fix: `--vh-full` = empty, shell height = auto
- After fix: `--vh-full` = `100vh`, shell height = `1028px` (1080 - 52), sidebar height = `1028px`, sidebar nav = `996px` with `overflow-y: auto`

**Tests:** 396/396 unit tests green. PM2 restarted on :8888.

**Glen confirmed:** Sidebar now scrolls. Fix verified in production.

## Entry 2: Scroll architecture redesign (permanent fix)

**Problem:** 30+ recurrent scroll bugs caused by two competing scroll strategies running simultaneously — CSS flex chain (`flex: 1; overflow-y: auto`) and JS pixel overrides (`fixScrollHeights()` + per-view rAF height hacks). The JS workarounds were layered on because the CSS chain was broken by the `--vh-full` bug. Now that `--vh-full` is fixed, the workarounds fight the working CSS.

**Solution:** Deleted all JS scroll workarounds (85 lines), added 8 lines of CSS. One pattern everywhere: parent `overflow: hidden`, fixed children `flex-shrink: 0`, scroll child `flex: 1; min-height: 0; overflow-y: auto`.

**Changes (commit 8bc59db):**
- Deleted `fixScrollHeights()` function and all 3 call sites
- Deleted per-view height hacks in renderTaskView, renderDashboard, renderCommandCentre
- Deleted `.pf` height recalc from resize listener
- Added `:has()` rules for split-pane views (tasks-view, pf, cc-page)
- Fixed `.tasks-view` and `.pf`: `flex: 1` instead of `height: 100%`
- Added `flex: 1; min-height: 0` to `.cc-page`

## Entry 7: Non-won leads appearing in dropdowns — FIXED

**Glen's report:** Lead-only clients (Anser, HHS, Morgan Stanley, Prince George County, Social Security Admin) were showing up in all client dropdowns throughout the app, not just on the Leads and Finances pages where they belong.

**Root cause:** `/api/clients` returned ALL rows from the `clients` table. `_apiClientsCache` stored them all. `getAllClients()` and `getContractedClients()` iterated the cache and included every entry — no check for whether the client had actual work or only non-won leads.

**Data investigation:** 13 clients total: 7 with tasks (active work), 5 with only non-won leads, 1 orphan ("test"). Pipeline stages confirmed: "Won" stage has `is_won = true`.

**Fix:**
- **Server** (`routes/clients.js`): Added `has_active_work` computed boolean to `/api/clients`. True when client has tasks OR at least one lead in a `is_won = true` stage.
- **Frontend** (`nbi_project_dashboard.html`): `getAllClients()` and `getContractedClients()` now filter `_apiClientsCache` by `has_active_work`.
- **Unaffected pages:** Leads page (uses `/api/clients` directly), Finances page (uses `_apiClientsCache` directly), Client portal (scoped by client ID).

**Glen's clarification:** The trigger for treating a lead as a client should be SOW received/uploaded. The existing "Won" pipeline stage (`is_won = true`) already represents this moment.
