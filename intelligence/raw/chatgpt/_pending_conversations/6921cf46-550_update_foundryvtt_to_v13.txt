- **Examples from the common 5e ecosystem**

Conclusion from the snapshot: the *official* 5e stack and actively-maintained 5e modules are targeting ÔÇťv13 as a wholeÔÇŁ, not particular sub-builds. 13.351 is within all of those declared compatibility ranges.

### 3. Cost / benefit of updating to 13.351

**Benefits**

1. **Security hardening**  
   - 13.351 explicitly closes two vulnerabilities that allowed the possibility of remote code execution (RCE) via edge-case document modification workflows. Foundry themselves recommend updating if your server is publicly accessible. ţłÇciteţłéturn1view0ţłü  
   - If you port-forward, run on a VPS, or use a host where strangers can hit your Foundry instance, this is a strong reason to upgrade.

2. **Stability and quality-of-life fixes**  
   - Fixes around Token HUD handling of ÔÇťorphanÔÇŁ tokens, the combat turn marker, and scene placeables are all in 13.351. ţłÇciteţłéturn1view0ţłéturn4view0ţłü  
   - Earlier maintenance patches in 13.350 fixed a raft of annoying bugs (macro bar glitches, token config preview issues, ownership visibility, drawing tools, etc.), again without major API changes. ţłÇciteţłéturn4view0ţłü  

3. **Alignment with the current 5e system**  
   - The official `dnd5e` systemÔÇÖs current branch is literally specified as ÔÇťFoundry Version 13.347+ÔÇŁ, so if you want its latest bug fixes and content, you are supposed to be on a late v13 build anyway. ţłÇciteţłéturn3view0ţłü  

**Risks**

1. **Module friction**  
   - Any update can shake out modules that are doing unsupported things, especially in the UI (ApplicationV2 etc.). Most of that pain, though, already happened when people jumped from v12 to the *first* v13 stable. Stable 9 is a small patch on top, not a breaking change.
   - Modules that *only* declare support up to 13.34x or 13.346 are not automatically broken by 13.351, they just have not been *re-verified* yet.

2. **Timing with your games**  
   - If you have a session tonight and zero time to test, then even a low-risk patch is badly timed. The main risk is scheduling, not the patch content itself.

My assessment, purely on the technical side:

- For **publicly accessible** or **hosted** servers:  
  Ôćĺ **Yes, I would update to 13.351**, because of the explicit RCE fixes.  
- For **purely local, LAN-only** use with no external exposure and everything currently working:  
  Ôćĺ It is still worth doing, but you can **schedule it for a quiet evening** and treat it as a low-risk, ÔÇťwhen convenientÔÇŁ maintenance step rather than urgent.

### 4. Suggested upgrade path with minimal guff

To keep friction down, I would do it like this.

#### 4.1 Backups and recovery (non-negotiable)

Before you touch the updater:

1. **Copy your user data**
   - Close Foundry.  
   - Copy the entire `Data` folder (or your equivalent user data path) to an external drive or a `Data_backup_YYYYMMDD` folder.

2. **Export key compendia**
   - In each world, export any critical compendium packs to JSON so you have standalone copies, particularly homebrew content.  

3. **Record the module set and versions**
   - In the Module Management screen, scroll through enabled modules and either:
     - Take a couple of screenshots, or  
     - Copy the list of enabled module IDs into a text file.  
   - This makes it easy to re-enable the same set, or to selectively disable a problem module later.

4. **Rollback plan**
   - Know how you would reinstall the *previous* core version from your Foundry account download page and then drop your backup `Data` folder back in if things go sideways.

#### 4.2 Update and test in a controlled way

1. **Update the core to 13.351**
   - Use the in-app updater or reinstall 13.351 from the official download as appropriate to your install type. ţłÇciteţłéturn1view0ţłéturn4view0ţłü  

2. **Update systems and key modules**
   - From the Setup screen, run ÔÇťCheck for UpdatesÔÇŁ on:
     - `dnd5e` system (bring it to at least 5.0.4 or ideally 5.1.x). ţłÇciteţłéturn3view0ţłü  
     - Core automation modules like Midi-QOL, Automated Conditions 5e, system customiser, and any others that *say* they support v13. ţłÇciteţłéturn2search6ţłéturn2search15ţłéturn2search8ţłü  

3. **Smoke-test with modules disabled first**
   - Launch a **copy** of your main world with **all add-on modules disabled**.
   - Confirm:
     - The world loads.  
     - You can open a character sheet, roll a couple of attacks/saves.  
     - No errors appear in the console.

4. **Re-enable modules in batches**
   - Start with ÔÇťinfrastructureÔÇŁ modules (e.g. compendium browsers, calendar, quality-of-life UI).
   - Then add combat/automation modules (Midi-QOL etc.) in a second batch.
   - After each batch, reload the world and run a tiny ÔÇťtest combatÔÇŁ to confirm rolls and effects work.

If something breaks, use the standard conflict-isolation trick: disable everything except system + hard dependencies, then re-add modules in small groups until you identify the culprit.
