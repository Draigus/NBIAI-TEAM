CREATE TABLE agg_kpi_weekly AS

       date_trunc('week', kpi_date)::date AS week_start,
       SUM(installs) AS installs,
       SUM(dau) AS dau_sum,

CREATE VIEW vw_metrics_daily AS
SELECT k.title_id, k.storefront_id, k.region_id, k.kpi_date AS date,
       k.installs, k.dau, k.wau, k.mau, k.sessions,
       k.payers, k.purchases, k.reactivations,
       r.net_usd,
       CASE WHEN k.dau > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.dau END AS arpdau_usd,
       CASE WHEN k.payers > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.payers END AS arppu_usd,
       CASE WHEN k.dau > 0 AND k.payers IS NOT NULL THEN k.payers::NUMERIC / k.dau END AS conversion_rate,
       k.churn_rate,
       k.d1_retention, k.d7_retention, k.d30_retention,
       k.is_proxy OR r.is_proxy AS any_proxy,
       k.method_id AS kpi_method_id,
       r.method_id AS revenue_method_id
FROM fact_kpi_daily k
LEFT JOIN (
  SELECT title_id, storefront_id, region_id, rev_date,
         SUM(net_usd) AS net_usd,
         BOOL_OR(is_proxy) AS is_proxy,
         MIN(method_id) AS method_id
  FROM fact_revenue_daily
  GROUP BY 1,2,3,4
) r
ON r.title_id = k.title_id AND r.storefront_id = k.storefront_id AND r.region_id = k.region_id AND r.rev_date = k.kpi_date;

Retention, units, and calculation rules
- DAU, WAU, MAU are unique user counts for the date or rolling windows. When derived, tag is_proxy and set method_id to the curve used.
- ARPDAU is net_usd divided by DAU for the same date and scope.
- ARPPU is net_usd divided by payers.
- Conversion rate is payers divided by DAU.
- Churn rate daily is estimated as max(0, (DAU[t−1] + NUU[t−1] − DAU[t] − reactivations[t]) ÷ DAU[t−1]) when telemetry is absent.
- Retention table stores cohort_date, age_days, retained_users. D1, D7, D30 are provided in fact_kpi_daily as convenience fields sourced from the cohort table.

Metrics dictionary seed
- dau users per day. lineage fact_kpi_daily.dau. proxy flag carried from method_id.
- payers users who completed ≥1 purchase on date. lineage fact_kpi_daily.payers.
- net_usd usd net receipts after discounts, refunds, taxes, and platform fees. lineage fact_revenue_daily.net_usd.
- arpdau_usd net_usd ÷ dau. lineage vw_metrics_daily. unit USD per user.
- arppu_usd net_usd ÷ payers. unit USD per payer.
- installs new device or account installs on date. lineage fact_kpi_daily.installs.
- reactivations users returning after being inactive for at least N days. lineage fact_kpi_daily.reactivations.
- d1_retention, d7_retention, d30_retention cohort-based ratios. lineage fact_retention_cohort_daily.

Partitioning and retention guidance
- Partition fact_kpi_daily, fact_revenue_daily, fact_rank_daily, fact_price_daily, fact_ua_spend_daily, fact_forecast_daily by date. Sort by title_id, date.
- Keep daily facts at full fidelity for 24 months, then archive to monthly aggregates for long-range modelling.
- Keep forecast artefacts for all model_version values for auditability.

# 1. Schema overview

## 1.1 Common conventions
- Keys: `title_id`, `storefront_id`, `region_id`, `date` fields are UTC dates.  
- Money: `*_local` in local currency, `*_usd` in USD. FX from `cfg_currency_rate`.  
- Governance columns on every fact: `source_system`, `batch_id`, `ingested_at_utc`, `updated_at_utc`.  
- Proxy lineage: `method_id` plus `is_proxy` boolean.  
- Partitions: all daily facts partitioned by date. Sort keys: `title_id, date`.

## 1.4 Materialised aggregates and standard KPI view
```sql
CREATE TABLE agg_kpi_weekly AS
SELECT title_id, storefront_id, region_id,
       date_trunc('week', kpi_date)::date AS week_start,
       SUM(installs) installs,
       SUM(nuu) nuu,
       SUM(dau) dau_sum,
       MAX(wau) wau,
       MAX(mau) mau,
       SUM(payers) payers,
       SUM(purchases) purchases,
       AVG(d1_retention) d1_retention,
       AVG(d7_retention) d7_retention,
       AVG(d30_retention) d30_retention
FROM fact_kpi_daily
GROUP BY 1,2,3,4;

CREATE TABLE agg_revenue_monthly AS
SELECT title_id, storefront_id, region_id, currency_id,
       date_trunc('month', rev_date)::date AS month_start,
       sku_type,
       SUM(gross_local) gross_local,
       SUM(discounts_local) discounts_local,
       SUM(refunds_local) refunds_local,
       SUM(taxes_local) taxes_local,
       SUM(platform_fees_local) platform_fees_local,
       SUM(net_local) net_local,
       SUM(gross_usd) gross_usd,
       SUM(net_usd) net_usd,
       SUM(units) units,
       SUM(payers) payers
FROM fact_revenue_daily
GROUP BY 1,2,3,4,5,6;

CREATE VIEW vw_metrics_daily AS
SELECT k.title_id, k.storefront_id, k.region_id, k.kpi_date AS date,
       k.installs, k.nuu, k.dau, k.wau, k.mau, k.sessions,
       k.payers, k.purchases, k.reactivations,
       r.net_usd,
       CASE WHEN k.dau > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.dau END AS arpdau_usd,
       CASE WHEN k.payers > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.payers END AS arppu_usd,
       CASE WHEN k.dau > 0 AND k.payers IS NOT NULL THEN k.payers::NUMERIC / k.dau END AS conversion_rate,
       CASE WHEN k.mau > 0 AND k.dau IS NOT NULL THEN k.dau::NUMERIC / k.mau END AS sticky_factor,
       k.churn_rate, k.d1_retention, k.d7_retention, k.d30_retention,
       k.is_proxy OR r.is_proxy AS any_proxy,
       k.method_id AS kpi_method_id,
       r.method_id AS revenue_method_id
FROM fact_kpi_daily k
LEFT JOIN (
  SELECT title_id, storefront_id, region_id, rev_date,
         SUM(net_usd) AS net_usd,
         BOOL_OR(is_proxy) AS is_proxy,
         MIN(method_id) AS method_id
  FROM fact_revenue_daily
  GROUP BY 1,2,3,4
) r
  ON r.title_id=k.title_id AND r.storefront_id=k.storefront_id
 AND r.region_id=k.region_id AND r.rev_date=k.kpi_date;
```

# 2. Metrics dictionary

Each entry includes name, unit, grain, definition, formula, lineage, null rules, and example SQL when derived.

**Installs**  
- Unit users. Grain daily per title, storefront, region.  
- Definition new device or account installations recorded or estimated for date d.  
- Lineage `fact_kpi_daily.installs`. Proxy when `is_proxy = true`.  
- Null rules store null when unknown.  
- Notes used as cohort seed for retention.

**NUU**  
- Unit users. Grain daily.  
- Definition new unique users who were active on date d.  
- Lineage `fact_kpi_daily.nuu`.

**DAU, WAU, MAU**  
- Unit users. Grain daily for DAU, rolling 7 for WAU, rolling 30 for MAU.  
- Lineage `fact_kpi_daily`.  
- Proxy tag carried from `method_id`.  
- Example null clamp set to null if negative after estimation.

**Sessions**  
- Unit sessions. Grain daily.  
- Lineage `fact_kpi_daily.sessions`. Optional if unavailable.

**Payers**  
- Unit users. Grain daily.  
- Definition unique users with at least one purchase on date d.  
- Lineage `fact_kpi_daily.payers`.

**Purchases**  
- Unit transactions. Grain daily.  
- Lineage `fact_kpi_daily.purchases`.

**Reactivations**  
- Unit users. Grain daily.  
- Definition users returning after inactivity threshold N days.  
- Lineage `fact_kpi_daily.reactivations`.

**Churn rate**  
- Unit ratio. Grain daily.  
- Definition estimated share of users who became inactive on date d.  
- Formula when proxy  
  `max(0, (DAU[d-1] + NUU[d-1] - DAU[d] - reactivations[d]) ÷ nullif(DAU[d-1],0))`.  
- Lineage `fact_kpi_daily.churn_rate`.

**D1, D7, D30 retention**  
- Unit ratio. Grain daily convenience fields.  
- Definition cohort retention for cohorts with `cohort_date = d-1`, `d-7`, `d-30`.  
- Preferred computation from `fact_retention_cohort_daily`.

**Gross revenue, Net revenue**  
- Unit USD or local. Grain daily by `sku_type`.  
- Lineage `fact_revenue_daily.gross_*`, `net_*`.  
- Net formula `gross − discounts − refunds − taxes − platform_fees`.

**Units**  
- Unit units. Grain daily by `sku_type`.  
- Definition units sold of base or DLC; pseudo units for IAP if configured.

**ARPPU**  
- Unit USD per payer. Grain daily.  
- Formula `net_usd ÷ payers`.  
- SQL
```sql
SELECT date, CASE WHEN payers>0 THEN net_usd/payers END AS arppu_usd
FROM vw_metrics_daily
WHERE title_id = :t AND storefront_id=:s AND region_id=:r;
```

**ARPDAU**  
- Unit USD per user. Grain daily.  
- Formula `net_usd ÷ DAU`. See view above.

**Conversion rate**  
- Unit ratio. Grain daily.  
- Formula `payers ÷ DAU`.

**Sticky factor**  
- Unit ratio. Grain daily.  
- Formula `DAU ÷ MAU`.

**DLC attach rate**  
- Unit ratio. Grain daily.  
- Definition estimated share of base-game MAU that owns DLC X.  
- SQL example
```sql
SELECT d.date,
       CASE WHEN m.mau>0 THEN dlc.units_cum::NUMERIC / m.mau END AS dlc_attach
FROM (
  SELECT rev_date AS date, SUM(units) OVER(ORDER BY rev_date ROWS UNBOUNDED PRECEDING) units_cum
  FROM fact_revenue_daily
  WHERE title_id=:t AND sku_type='dlc'
) dlc
JOIN (
  SELECT date, mau FROM vw_metrics_daily WHERE title_id=:t
) m ON m.date=dlc.date;
```

**UA metrics: Impressions, Clicks, Installs, Spend, CPM, CPC, CPI**  
- Units impressions, clicks, installs, USD, USD per thousand, USD per click, USD per install.  
- Grain daily by title, region, channel.  
- Lineage `fact_ua_spend_daily`. CPM, CPC, CPI are generated columns in table.

**ROAS d, ROAS 7, ROAS 30**  
- Unit ratio.  
- Definition cumulative revenue from attributed cohorts divided by spend after d, 7, or 30 days.  
- Example SQL sketch joining attributed installs to revenue is model dependent. Store as model output in a view `vw_roas_*` in Foresight.

**Forecast ribbons**  
- Unit metric units. Grain daily by metric and scenario.  
- Lineage `fact_forecast_daily.point, low, high`.  
- Backtest error metrics in `fact_forecast_backtest` with `mape, mae, rmse`.

**Rev rec rollforward**  
- Bookings, Recognized, Deferred open, Deferred close in `fact_revrec_daily`.  
- Example rollforward check  
```sql
SELECT revrec_date, deferred_open_usd + bookings_usd - recognized_usd AS deferred_close_check
FROM fact_revrec_daily
WHERE title_id=:t AND sku_id=:sku
ORDER BY revrec_date;
```

# 4. Retention, partitioning, and SLAs

- Fact tables keep 24 months at daily grain online. Monthly aggregates retained for life.  
- Partitions by date, sort by `title_id, date`.  
- Ingest SLA daily at 09:00 UTC. Forecast refresh after ingest.  
- FX and fee schedules versioned by effective dates.  
- Data quality checks  
  1. Non negative: installs, dau, payers, units, gross.  
  2. Bounded ratios 0 to 1: retention, conversion_rate, sticky_factor.  
  3. Revenue reconciliation per day: `net_local = gross_local − discounts_local − refunds_local − taxes_local − platform_fees_local`.  
  4. Rev rec identity: `deferred_close = deferred_open + bookings − recognized`.  
  5. Proxy flag required when `method_id` belongs to non-telemetry methods.

# 5. Example queries

**Daily KPI pack**
```sql
SELECT * FROM vw_metrics_daily
WHERE title_id=:t AND storefront_id=:s AND region_id=:r
  AND date BETWEEN :d0 AND :d1
ORDER BY date;
```

**Price elasticity what-if**
```sql
SELECT p.price_date,
       p.live_price_local,
       e.elasticity,
       k.dau,
       (1 + e.elasticity * p.discount_pct) * COALESCE(k.dau,0) AS demand_index
FROM fact_price_daily p
LEFT JOIN fact_kpi_daily k
  ON k.title_id=p.title_id AND k.storefront_id=p.storefront_id
 AND k.region_id=p.region_id AND k.kpi_date=p.price_date
JOIN cfg_price_elasticity e ON e.sku_type = 'base'
WHERE p.title_id=:t;
```

**ROAS 7 sketch**  
Assumes an attribution view `vw_attr_installs` that maps installs to revenue by cohort and day.
```sql
SELECT a.title_id, a.region_id, a.channel_id, a.install_date,
       SUM(r.net_usd) / NULLIF(SUM(a.spend_usd),0) AS roas_7
FROM vw_attr_installs a
JOIN fact_revenue_daily r
  ON r.title_id=a.title_id AND r.region_id=a.region_id
 AND r.rev_date BETWEEN a.install_date AND a.install_date + INTERVAL '7 days'
GROUP BY 1,2,3,4;
```

# Critique summary
The first cut is close, but several issues would slow modelling and QA in practice: revenue lacks SKU-level fidelity, geo is too coarse for iOS/Google, metrics codes are free-text, and data quality governance is implicit rather than explicit. Forecast artefacts are fine, but we should add model registry and metric locks to prevent silent drift. Price, release, and events need a shared calendar of “drivers” to improve attribution.

# Critic matrix and fixes
Issue | Impact | Fix applied
:--|:--|:--
Revenue fact keyed by `sku_type` only | Cannot analyse DLCs, seasons, and bundles at SKU resolution; hampers rev rec joins | Replace with SKU-level revenue fact; keep a view for `sku_type` rollups
Region too coarse, no country | Rank and featuring datasets are country-level; proxy mapping loses fidelity | Add `dim_country` and optional `country_code` on facts; keep `region_id` as rollup
Free-text metric codes | Risk of typos breaking joins and views | Add `dim_metric` and foreign keys from forecast and views
No explicit method precedence or source availability | Hard to reason about proxy vs direct and conflict resolution | Add `meta_source_availability` and `meta_method_precedence`
No driver calendar | Weak attribution and scenario wiring | Add `dim_driver_event` with daily fact for on/off drivers
DQ and load audits implicit | Failures surface late, lineage unclear | Add `dq_check_catalog`, `dq_result_daily`, and `etl_load_audit`
Retention convenience only | Cohort LTV lacks a home | Add `fact_ltv_cohort_daily` for forecast backtests and scenario LTV
Enums in wide tables | Harder to extend | Introduce lightweight dims: `dim_business_model`, `dim_release_type`, `dim_chart_type`
Ratio bounds not enforced | Data drift | Add `CHECK` constraints on 0..1 fields

## Daily KPI pack by SKU type and country rollup
```sql
CREATE VIEW vw_metrics_daily AS
SELECT k.title_id, k.storefront_id, k.region_id, k.country_id,
       k.kpi_date AS date,
       k.installs, k.nuu, k.dau, k.wau, k.mau, k.sessions,
       k.payers, k.purchases, k.reactivations,
       r.net_usd,
       CASE WHEN k.dau > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.dau END AS arpdau_usd,
       CASE WHEN k.payers > 0 AND r.net_usd IS NOT NULL THEN r.net_usd / k.payers END AS arppu_usd,
       CASE WHEN k.dau > 0 AND k.payers IS NOT NULL THEN k.payers::NUMERIC / k.dau END AS conversion_rate,
       CASE WHEN k.mau > 0 AND k.dau IS NOT NULL THEN k.dau::NUMERIC / k.mau END AS sticky_factor,
       k.churn_rate, k.d1_retention, k.d7_retention, k.d30_retention,
       k.is_proxy OR r.is_proxy AS any_proxy,
       k.method_id AS kpi_method_id, r.method_id AS revenue_method_id
FROM fact_kpi_daily k
LEFT JOIN (
  SELECT title_id, storefront_id, region_id, country_id, rev_date,
         SUM(net_usd) AS net_usd,
         BOOL_OR(is_proxy) AS is_proxy,
         MIN(method_id) AS method_id
  FROM fact_revenue_daily
  GROUP BY 1,2,3,4,5
) r
ON r.title_id=k.title_id AND r.storefront_id=k.storefront_id
AND r.region_id=k.region_id AND COALESCE(r.country_id,0)=COALESCE(k.country_id,0)
AND r.rev_date=k.kpi_date;
```

# Metrics dictionary adjustments

Metric | Update
:--|:--
Net revenue | Clarified SKU lineage. Now sourced from `fact_revenue_daily.net_usd` at SKU granularity, rolled up by `vw_revenue_by_type_daily`.
ARPDAU, ARPPU | Unchanged formulas. Document that country-level joins take precedence when present to match rank-derived proxies.
DLC attach | Prefer SKU-level cumulative owners from `sku_id` units streams. Roll up to type only if SKU missing.
Retention | Cohort logic unchanged. LTV for cohorts now lives in `fact_ltv_cohort_daily`.
ROAS x | Defined as model output view joining `fact_ua_spend_daily` to attributed cohort revenue. Not prescribed in base schema due to method variance; hook through `model_registry`.

# Governance and performance

Topic | Decision
:--|:--
Partitioning | All daily facts partitioned by date. Secondary sort keys `title_id, region_id, country_id`.
Retention | Two years daily hot, life-of-project monthly aggregates.
FX and fees | Versioned by effective dates. Require a row on load day; fail the load if absent for referenced currency.
DQ checks | Seed `dq_check_catalog` with bounds on ratios, non-negative money, rev rec identity, and FX coverage.
Lineage | Require `method_id` and `is_proxy` on all facts. Populate `meta_source_availability` per title and metric.

  CASE WHEN k.payers>0 AND r.net_usd IS NOT NULL THEN r.net_usd/k.payers END AS arppu_usd,
