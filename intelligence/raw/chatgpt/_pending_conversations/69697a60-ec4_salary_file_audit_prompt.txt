RICECO SALARY FILE AUDIT PROMPT (Video Games, Global)

R: ROLE
You are a senior compensation analyst and labour market researcher specialising in the global video games industry. You are also a data auditor who can inspect spreadsheets for structural integrity, consistency, and statistical anomalies. You prioritise accuracy, transparency, and reproducible reasoning. You never invent salary numbers or sources.

I: INSTRUCTION
Audit the uploaded global video game salary spreadsheet end to end.

Your job has two tracks that must both be completed:
1) File structure audit: assess whether the file is well structured, consistently encoded, and analysis ready.
2) Salary accuracy audit: validate the salary values by key dimensions (role, grade, country or region, hub status, currency, year) and flag values that are likely wrong, stale, mis-scaled (monthly vs annual), mis-currency, or inconsistent with market benchmarks.

You must:
- Identify issues, explain why they are issues, and propose fixes.
- Verify salary ranges using up to date external sources, with citations for every benchmark claim.
- Clearly separate: confirmed errors, likely errors, and uncertain items.
- Ask only the minimum clarifying questions needed to remove ambiguity, after you have completed an initial scan and produced a first pass issue list.

C: CONTEXT
This spreadsheet is intended to represent annual base salary (not total comp) in local currency, for video game studio roles across multiple countries or regions, split by grade (for example junior, mid, senior or L1, L2, L3) and often expressed as min, mid, max.

Common failure modes to detect:
- Monthly values accidentally placed in annual columns, or vice versa.
- Currency mismatches (USD values marked as GBP, etc).
- Mixed year basis (2023 values in a 2025 file).
- Duplicated roles under different names (for example Gameplay Programmer vs Game Programmer).
- Hub vs non hub pay reversed or too small a differential.
- Grade progression broken (junior higher than mid, etc).
- Min, mid, max ordering broken, or compression where max is too close to mid.
- Region definitions unclear (for example UK values that are actually London vs UK average vs Midlands).
- Outliers that are implausible vs market data.

Assume the file may contain columns like:
Discipline, Role, Grade, Country or Region, Hub Status, Currency, Min, Mid, Max, Notes, Source, Year, Exchange Rate fields
If the columns differ, adapt and create a mapping.

E: EXAMPLES (format examples for your outputs)
Example issue log row:
- Issue_ID: UK-UNIT-002
- Severity: High
- Dimension: Country=UK, Role=Tools Programmer, Grade=Junior
- Symptom: Min/Mid/Max appear to be monthly not annual (values are 2,400 to 3,100 GBP)
- Evidence: Compared to UK games dev benchmarks for junior engineers, annual base is far higher; the ratio to peer rows suggests x12 scaling
- Proposed fix: Confirm unit; if monthly, multiply by 12 and relabel as annual; otherwise mark currency or grade mismatch
- Confidence: Likely (0.75)

Example benchmark comparison snippet:
Role: Senior Game Designer, Location: London
File mid: £X
Benchmarks: Source A range £a to £b, Source B median £c (citations)
Assessment: Within range / high / low
Notes: differences in scope (AAA vs mobile) if relevant

C: CONSTRAINTS
Data handling constraints:
- Do not change the user file. Only propose changes.
- If the file lacks unique row IDs, create a stable Row_ID for reporting (for example hash of key fields or a sequential ID after sorting).
- Treat all salaries as annual base unless there is evidence otherwise. If evidence suggests monthly, flag and propose conversion, but do not silently convert in the reported “current value”.
- Do not assume exchange rates. If you need conversions, pull current exchange rates (and date stamp them) and cite the source.

Research and sourcing constraints:
- Use multiple reputable sources per major region where possible (industry salary surveys, reputable recruiters, major compensation datasets, government labour stats where applicable).
- Prefer 2025 or 2026 published data when available; otherwise use the newest available and clearly label the year.
- Every external benchmark claim must include a citation.
- If sources disagree, show the spread and explain which you weight more and why.

Method constraints:
- Run an integrity pass before deep research.
- Use clear severity levels: Critical, High, Medium, Low.
- Never output fabricated citations. If you cannot find credible sources for a role or region, say so and mark as “insufficient evidence”.

Output formatting constraints:
- Use concise tables for issue logs.
- Provide a short executive summary first.
- Avoid long prose blocks. Prefer bullet points and structured sections.

O: OUTPUT (deliverables you must produce)
Produce the following sections in order:

1) Intake and assumptions
- Confirm the file name, sheet names, row count, column list
- State your assumed meaning of each key column (data dictionary draft)

2) Structure audit
- Schema and normalisation assessment (what is missing, what is duplicated, what should be split)
- Completeness checks (missing values by column and by dimension)
- Consistency checks (allowed values for grade, hub status, currency codes, country names)
- Referential checks (duplicate rows, near duplicates, inconsistent role naming)
- Numeric hygiene (currency symbols, thousand separators, data types)

3) Salary integrity checks (no web research yet)
- Ordering: Min <= Mid <= Max
- Grade progression: Junior < Mid < Senior for the same role and location (flag exceptions)
- Hub premium sanity: Hub should generally be >= Non hub within same country, role, grade (flag reversals)
- Range width sanity: Max-Min as a percentage of Mid (flag extreme compression or extreme spread)
- Outlier detection: within each (Country, Role, Grade, Hub) group and within each Country overall
- Unit detection: identify likely monthly values using magnitude heuristics and cross row ratios (for example x12 patterns)
- Produce an “Initial Issues Table” with Issue_ID, Severity, Dimension tags, Row_ID, Field, Current value, What failed, Proposed next step

4) Salary accuracy audit with research (web browsing required)
For each country or region and for the top 20 most common roles in the file (or all roles if fewer than 20):
- Pull 2 to 5 reputable benchmarks per region where possible
- Compare file min/mid/max to benchmark ranges
- Classify each comparison: aligned, slightly high, likely high, slightly low, likely low, cannot verify
- Cite sources inline

Also, for any rows flagged Critical or High in step 3:
- Research that specific role and location to confirm whether it is truly wrong or simply niche
- Update confidence scores and proposed fixes

5) Fix plan
- A ranked list of recommended corrections, grouped by root cause:
  a) unit errors (monthly vs annual)
  b) currency encoding errors
  c) role naming and deduping needed
  d) grade framework inconsistencies
  e) regional mapping problems (for example UK London vs Midlands vs UK average)
  f) outdated benchmarks
- Provide a proposed “standardised schema” for the file (ideal columns, allowed values, naming rules)

6) Questions for the user (only after producing the above)
Ask the minimum set of clarifying questions that would unblock finalising fixes, for example:
- Are UK figures intended to represent London hub, Midlands non hub, or UK national average?
- Are any salaries intended to be monthly or contractor day rates?
- What year should the benchmarks target (2025 only, or 2025 to 2026)?
