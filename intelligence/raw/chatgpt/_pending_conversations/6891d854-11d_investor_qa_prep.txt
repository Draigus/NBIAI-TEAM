**CRUD = Create / Read / Update / Delete.**  
When we say “**CRUD player-profile**” for SARGE’s backend vertical slice, we mean the very first set of API endpoints (or gRPC calls) that let the game and the Telegram mini-app do the four basic things every service must support for a player’s record:

| Operation | Example in SARGE | Why it matters early |
|-----------|-----------------|----------------------|
| **Create** | `POST /player` takes the user’s Telegram ID, chosen callsign and region, returns a new `player_id`. | Proves account-creation flow, links TG Single-Sign-On to our DB. |
| **Read**  | `GET /player/{id}` returns profile JSON → username, level, medals, cosmetic load-out, settings. | Lets the client render the lobby and persist progress between sessions. |
| **Update** | `PATCH /player/{id}` to change avatar, equip a new skin, increment XP after a match, toggle push-notifications, etc. | Validates our write paths and real-time cache invalidation. |
| **Delete** | `DELETE /player/{id}` (or soft-delete flag) for GDPR/account-wipe requests. | Needed for compliance and to test data‐retention logic. |

Building this slice first gives us:

1. A **single source of truth** for identity and progression before we bolt on matchmaking, inventory, battle-pass, etc.  
2. A way to performance-test the full stack (API → auth → DB → cache) under load with dummy users.  
3. A concrete contract for the Unity client and the future live-ops tools to integrate against, so the rest of the backend work can proceed in parallel.

Once the CRUD profile service is solid, we layer in session matchmaking and the authoritative game-state loop for 1-v-1 combat.
