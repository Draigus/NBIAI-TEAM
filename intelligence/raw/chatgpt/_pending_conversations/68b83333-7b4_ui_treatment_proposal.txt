# Audit of the first prompt

1) Ambiguous stage vs probability  
- You defined both likelihood_enum and status_enum with overlapping values like “Signed.” That creates confusion for filters and charts. The spreadsheet clearly treats “Signed” as a stage and reserves “High, Favourable, Medium, Low” for probability. These must be separate fields with a single source of truth and a mapping between them for weighting.

2) Weighting rules are under-specified  
- You set global weights but did not define how weights interact with stage. For example, “On Hold” should zero out weighted pipeline even if likelihood is “High.” The rules need explicit precedence and defaults.

3) Revenue recognition is too naïve  
- The formula “ROM divided by weeks if no staffing” hides big error bars. You need three recognition modes: by assignments hour-rate, by even spread over duration, and by custom schedule for fixed fee milestones. Store the chosen mode per contract.

4) ROM vs contract value  
- The spreadsheet column is ROM. Once the SoW is signed the value changes from ROM to Contract Value. The model should track both and keep an audit of the transition.

5) Missing contract types  
- T&M, Fixed Fee, Retainer. Calculations, run rate, and recognition differ. Not captured.

6) Signed & Paid view filter is brittle

8) Timeline data gaps

9) Currency, VAT, and multi-currency

10) Performance and scalability

## Weighting rules

Create settings with defaults:  
- weight_high=0.70, weight_favourable=0.60, weight_medium=0.40, weight_low=0.20.  
- Precedence: if stage in [On_Hold, Not_Doing, Delayed] then weight=0 regardless of likelihood. If stage in [Signed, In_Progress, Paid] then weight=1. Otherwise use likelihood weight.  
- WeightedValue(contract) returns valueForStage × weight, where valueForStage is contractValue if present else romValue.  

## Revenue recognition

Implement a helper that returns expected revenue per week:
- ASSIGNMENTS: sum(weeklyHours × hourlyRate) for each active week between start and end.  
- EVEN: even split of valueForStage over the active weeks.  
- CUSTOM: sum of RevSchedule amounts by week.  
- Determine start and end: start = estStartDate if set, else today. end = endDate if set, else start plus estWeeks. Ignore contracts without a resolvable date window in week-based projections, but include them in unweighted and weighted totals.
