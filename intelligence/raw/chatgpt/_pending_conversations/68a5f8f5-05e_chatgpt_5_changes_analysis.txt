Short version: you are not imagining it. Some changes can make a new model feel worse in real use, even if lab scores go up.

Why this happens, in practice:
- Tighter safety and brand constraints, more refusals or evasive wording, more generic answers.
- Preference tuning that rewards shortness and “polite neutrality,” which can blunt deep, specific reasoning.
- Compute budget per reply trimmed, so less internal deliberation and shallower planning.
- Training data shifts and de-dup to reduce memorization, which can lower recall on niche topics and older facts.
- Context handling quirks, including lost thread goals, weak instruction priority, or long chats exceeding useful window.
- Tool and browsing policies changing, which can add friction or prevent direct answers in sensitive domains.
- Guardrails against chain-of-thought leakage, so you see conclusions without the working, which feels dumber even when correct.
- Sampling defaults changed, so outputs look safer but also flatter.

How to get better results right now:
1. Front-load constraints in the first message: goals, audience, do and do not lists, success criteria, format, length.
2. Force assumptions to be stated: “List assumptions first, then solve, keep it under N bullets.”
3. Ask for a brief plan before execution: “Outline the steps, then do them.” This preserves depth without fluff.
4. Require verification: “Cite sources for any claim after June 2024, else say I don’t know.”
5. Lock style rules: “No em dashes, no icons, concise, numbered lists, active voice.”
6. Allow web search when freshness matters, forbid it when you want pure synthesis.
7. For complex tasks, request two passes: a draft, then a self-audit against your checklist.
