# Tenant-to-Tenant Migration (NSI → NBI) Tasks

## Preparing both tenants  
**Task: Inventory existing environments** – In the NSI and NBI tenants, list all users, groups, licenses, and roles. In the Microsoft 365 Admin Center (admin.microsoft.com), go to **Users > Active users** and export or document all accounts, licenses and admin roles. Do the same for NBI. Ensure no users are assigned critical features (like Exchange or SharePoint) that will block domain removal.  
**Task: Verify migration licensing** – Cross-tenant migrations require a per-user *Cross-Tenant User Data Migration* license【1†L89-L97】. Confirm that you have purchased this add-on license for every migrating user (all in NSI who will move content). Also check that you have sufficient Microsoft 365 licenses (E3/E5/F3, etc.) in the target (NBI) tenant for all users.  
**Task: Assign admin roles and permissions** – Assign Global Administrator and SharePoint Administrator roles to the migration operators in both tenants. These roles are needed for mailbox and OneDrive/SharePoint migrations【29†L61-L65】. In each tenant’s Admin Center, go to **Roles > Admin roles**, find “Global admin” and “SharePoint admin,” and add the migration team accounts.  

## Creating users in the target tenant and assigning licenses  
**Task: Create user accounts in NBI** – For each NSI user being migrated (Bryan, Tom, Jeff, Devin, Denise, Jessica) and for all new users (Patrice, Magnus, Glen, Nagesh, Andy, Yana), create corresponding user accounts in NBI. In the NBI Admin Center, go to **Users > Active users**, click **Add a user**【49†L82-L89】. In the setup wizard, fill in the user’s name and username (e.g. tom@example.com), choose the correct domain, and set a password.  
**Task: Assign licenses to target users** – During account creation or afterward, assign each user the necessary Microsoft 365 licenses and the migration license. When adding a user in the wizard, choose **Location** and select the appropriate license (e.g. Microsoft 365 E3) and enable required services【49†L92-L99】. Ensure each migrating user has an Exchange Online mailbox license and OneDrive plan. Also apply the Cross-Tenant migration license to all.  
**Task: Join users to groups** – If any migrating users need access to specific security or Microsoft 365 groups in NBI, add them now. For example, in NBI Admin Center go to **Groups > Active teams & groups**, open the appropriate group, and add the user under **Members**. Pre-create any “guest” accounts: if an NSI user will **not** move but needs continued access (none in this list), add them as guest users in NBI.  

## Setting up cross-tenant migration trusts and endpoints  
**Task: Connect to each tenant via PowerShell** – On your workstation, open PowerShell and install/connect SharePoint and Exchange modules. Run `Connect-SPOService -Url https://<tenant>-admin.sharepoint.com` for NSI and then for NBI【29†L79-L87】. Also run `Connect-ExchangeOnline -UserPrincipalName admin@<tenant>` for NSI and NBI (using Exchange Online PowerShell). This establishes a session to each tenant’s admin endpoints.  
**Task: Configure SharePoint/OneDrive migration trust** – On **NSI (source)** SharePoint, run:  
```
Set-SPOCrossTenantRelationship -Scenario MnA -PartnerRole Target -PartnerCrossTenantHostUrl <NBI_CrossTenantHostUrl>
```  
Then on **NBI (target)** SharePoint, run:  
```
Set-SPOCrossTenantRelationship -Scenario MnA -PartnerRole Source -PartnerCrossTenantHostUrl <NSI_CrossTenantHostUrl>
```  
Each command must use the other tenant’s host URL obtained via `Get-SPOCrossTenantHostURL`. These PowerShell commands establish the cross-tenant trust【12†L110-L119】. (The NBI Global Admin will receive an email to approve the trust request.)  
**Task: Create Exchange migration app in target** – In the Azure portal for NBI, go to **Azure Active Directory > App registrations** and register a new application (e.g. “NSI-Migration-App”). Record its Application (client) ID and create a new client secret. Grant this app the **Exchange.ManageAsApp** permission and click **Grant admin consent**.  
**Task: Create migration endpoint in target Exchange Online** – In Exchange PowerShell (connected to NBI), run:  
```powershell
$AppId = "<AppID of migration app>"
$secret = "<secret value>"
$Credential = New-Object System.Management.Automation.PSCredential ($AppId,(ConvertTo-SecureString $secret -AsPlainText -Force))
New-MigrationEndpoint -Name "NSI-Endpoint" -RemoteServer outlook.office.com -RemoteTenant "<NSI_onmicrosoft_domain>" -Credentials $Credential -ExchangeRemoteMove:$true -ApplicationId $AppId
```  
This creates the cross-tenant migration endpoint【21†L300-L304】. Ensure the target delivery domain (RemoteTenant) matches NSI’s onmicrosoft.com domain.  
**Task: Create organization relationship (target side)** – Still in target Exchange PS, run:  
```powershell
New-OrganizationRelationship -Name "NSI-OrgRel" -Enabled $true -MailboxMoveEnabled $true -MailboxMoveCapability Inbound -DomainNames "<NSI_TenantID_GUID>"
```  
Replace `<NSI_TenantID_GUID>` with NSI’s Tenant GUID (not name). This enables incoming cross-tenant mailbox moves from NSI to NBI【21†L321-L329】.  
**Task: Consent migration app on source side** – In a browser, navigate to the URL given by your NBI admin (it will look like `https://login.microsoftonline.com/<NSI_onmicrosoft>.onmicrosoft.com/adminconsent?client_id=<AppId>&redirect_uri=https://office.com`). Sign in as NSI Global Admin and accept the app permission prompt. This grants the NBI migration app permission on NSI.  
**Task: Create organization relationship (source side)** – In NSI’s Exchange PowerShell, run (after adding a mail-enabled security group “Migrators” containing all migrating users):  
```powershell
New-OrganizationRelationship -Name "NBI-OrgRel" -Enabled $true -MailboxMoveEnabled $true `
  -MailboxMoveCapability RemoteOutbound -DomainNames "<NBI_TenantID_GUID>" `
  -OAuthApplicationId "<AppID>" -MailboxMovePublishedScopes "Migrators"
```  
Here `<NBI_TenantID_GUID>` is NBI’s Tenant GUID and `<AppID>` is the migration app’s ID. This enables NSI to send mailboxes to NBI and restricts moves to the “Migrators” group【21†L374-L382】. (Ensure the group’s recipients are exactly the users being migrated.)  
**Task: (Optional) Verify org relationships** – You can run `Get-OrganizationRelationship` in each Exchange Online PowerShell to confirm the relationships and MailboxMoveEnabled settings. 

## Migrating email, calendars, OneDrive, and SharePoint content  
**Task: Migrate Exchange mailboxes and calendars** – On NBI (target), prepare a CSV file (`users.csv`) listing the target UPN of each migrating user (e.g. `bryan@nbionmicrosoft.onmicrosoft.com`). Then in Exchange PowerShell or EAC, start a cross-tenant migration batch. For example, run:  
```powershell
New-MigrationBatch -Name NSI-to-NBI -SourceEndpoint NSI-Endpoint `
  -CSVData ([System.IO.File]::ReadAllBytes("users.csv")) -AutoStart `
  -TargetDeliveryDomain "<NBI_onmicrosoft_domain>"
```  
This initiates mailbox moves【26†L611-L619】. Alternatively, in the new Exchange Admin Center, go to **Migration > Migrate to Exchange Online > Cross-tenant migration**, and follow the wizard. Monitor the batch in Exchange PS (`Get-MigrationBatch`) or EAC until completion. After syncing, complete the batch (or it may auto-complete). Moved mailboxes will now contain all email, contacts, calendars, tasks and notes (Top of Information Store)【26†L678-L684】.  
**Task: Migrate OneDrive content** – For each user Bryan, Tom, Jeff, Devin, Denise, Jessica, run in NSI’s SPO PowerShell:  
```powershell
Start-SPOCrossTenantUserContentMove -SourceUserPrincipalName user@nsi.onmicrosoft.com `
  -TargetUserPrincipalName user@nbi.onmicrosoft.com `
  -TargetCrossTenantHostUrl https://<NBI>-my.sharepoint.com
```  
This command starts copying that user’s OneDrive from NSI to NBI【34†L75-L84】. Adjust UPNs to the real user accounts. You can batch up to 4,000 moves; to schedule for later, add `-PreferredMoveBeginDate` parameter. After starting, check progress with `Get-SPOCrossTenantUserContentMoveState -PartnerCrossTenantHostURL https://<NBI>-my.sharepoint.com` in NSI PowerShell【34†L141-L149】.  
**Task: Migrate SharePoint site content** – If any SharePoint sites must be migrated, run in NSI’s SPO PowerShell for each site:  
```powershell
Start-SPOCrossTenantSiteContentMove -SourceSiteUrl "https://<NSI>.sharepoint.com/sites/sitename" `
  -TargetSiteUrl "https://<NBI>.sharepoint.com/sites/newsitename" `
  -TargetCrossTenantHostUrl "https://<NBI>.sharepoint.com"
```  
This moves the entire site content【36†L81-L89】. For sites tied to Microsoft 365 Groups (e.g. Teams sites), use:  
```powershell
Start-SPOCrossTenantGroupContentMove -SourceGroupAlias OldTeam1 `
  -TargetGroupAlias NewTeam1 -TargetCrossTenantHostUrl "https://<NBI>.sharepoint.com"
```  
This migrates the group’s SharePoint files【36†L111-L119】. (Ensure the target site or group does **not** already exist—migration creates it.) Monitor with `Get-SPOCrossTenantSiteContentMoveState` or `Get-SPOCrossTenantGroupContentMoveState`.  

## Transferring custom domains and DNS records  
**Task: Remove custom domain from NSI** – Before moving the domain, change any user or group addresses using it. In NSI Admin Center, go to **Users > Active users** and edit each user to use the default `*.onmicrosoft.com` domain. Pay special attention to Global Admins (you may create a backup admin on onmicrosoft.com to avoid lockout). According to Microsoft, removing a domain does **not** delete data【51†L148-L153】, but you must sign users out and in under a different domain first. Once no accounts use the domain, go to **Settings > Domains**, select the domain, and click **Remove**. Follow the wizard to detach it.  
**Task: Add custom domain to NBI** – In NBI Admin Center, go to **Settings > Domains**, click the blue **Add domain** button【40†L101-L109】. Enter the domain name and click **Next**. Choose a verification method (TXT or domain connect) and complete it. Then for DNS setup, you can let Microsoft automatically configure (Domain Connect) or choose **“I’ll add the DNS records myself”**【40†L125-L134】. Add the provided MX record (to route mail to NBI), CNAME (Autodiscover), and other records at your DNS host. Finish the wizard to validate the domain.  
**Task: Update DNS as needed** – At your DNS registrar, ensure all records for email and services point to NBI. For example, update the MX record to NBI’s (as shown in the wizard), and update any SPF or TXT records. Also set the new domain as the default in NBI (Settings > Domains > make default).  
**Task: Verify domain transfer** – After a short propagation period, log into NBI Admin Center and confirm the domain’s status is “Setup complete.” Send a test email to each migrated user’s custom email (e.g. brian@yourdomain.com) to verify delivery to the NBI mailbox.  

## Reconfiguring apps and devices  
**Task: Update OneDrive sync clients** – In Windows, right-click the OneDrive cloud icon in the notification area and select **Help & Settings > Settings**. Under the **Account** tab, click **Add an account**【43†L158-L163】. Sign in with the user’s new NBI credentials (user@newdomain.com). This will reconfigure the user’s OneDrive sync to the new tenant. (They may need to re-share or migrate any unsynced files manually.)  
**Task: Reconfigure Outlook** – In Outlook desktop, go to **File > Account Settings > Email Accounts** (in **View settings** if using New Outlook【45†L290-L298】). Click **Add Account** and enter the user’s migrated email (for example, tom@newdomain.com). Complete the wizard (Outlook will auto-discover via Office 365). After confirming mail sync, remove the old NSI account profile if desired (under **Account Settings**, manage and remove the old account).  
**Task: Update Teams and Office sign-ins** – In Microsoft Teams, click the user’s profile picture and select **Sign out**. Then sign in again using the NBI account (you may need to “switch directories” to the new tenant). For Office apps (Word/Excel), open each app and go to **File > Account**; sign out of the old account and **Sign in** with the new NBI account. On mobile devices, have users remove their NSI accounts from Outlook, Teams, and OneDrive apps and add the new accounts.  
**Task: Re-add any third-party apps** – If users had single sign-on or other integrated apps (e.g. Adobe, Salesforce) tied to their NSI identity, update those apps to use the new tenant’s identity or credentials.  

## Validating success and decommissioning  
**Task: Verify mailbox and calendar moves** – In NBI, log into each migrated user’s Outlook Web Access and confirm that all emails, contacts, and calendar entries from NSI are present. Send test messages between migrated users and check calendar invites. Note: Teams meeting URLs do not migrate; notify users to recreate meetings if needed.  
**Task: Verify OneDrive and SharePoint data** – Have each user browse their new OneDrive and ensure files/folders match the old content. In SharePoint, navigate to migrated sites and check documents and permissions. Microsoft’s migration retains shared links by redirecting them【16†L62-L64】, so old SharePoint/OneDrive links (if any) should automatically redirect to the new tenant location. Test a shared link from an old document to confirm it opens in NBI.  
**Task: Remove migration endpoints and trusts** – Once all data is confirmed, clean up migration configuration. In Exchange PowerShell (NBI), run `Get-MigrationEndpoint` to list endpoints and then `Remove-MigrationEndpoint -Identity NSI-Endpoint` to delete it. Also run `Remove-OrganizationRelationship -Identity NSI-OrgRel`. In NSI, remove its “NBI-OrgRel” as well. This tidies up the cross-tenant relationships【26†L643-L650】. In SPO PowerShell, the `Set-SPOCrossTenantRelationship` has no remove command, but no further action is needed beyond closing sessions.  
**Task: Decommission old tenant** – Finally, in NSI Admin Center, you can disable or remove the migrated user accounts (or at least remove their licenses). Remove any custom domains (should already be done) and unused services. If NSI will be retired, close any billing accounts. Before full shutdown, ensure you have backups of any residual data.

### **BL-09  Install PowerShell modules locally**

```powershell
# NSI

# NBI
Connect-ExchangeOnline -UserPrincipalName migration.ops@nbi.onmicrosoft.com
Connect-SPOService -Url https://nbi-admin.sharepoint.com
```
**Done-when:** No error messages and each `Get-Mailbox` runs in the correct tenant.

### **BL-11  Register migration app in NBI**  
1. NBI portal.azure.com ➜ **Entra ID ▸ App registrations ▸ New registration**.

3. **Certificates & Secrets ▸ New client secret** ➜ copy Value.  
4. **API permissions ▸ Add permission ▸ APIs my org uses ▸ Office 365 Exchange Online**.  
5. **Application permissions ▸ Mailbox.Migration (plus Mailbox.ReadWrite)** ➜ **Add**.  
6. **Grant admin consent** ➜ Yes.  

Collect **Tenant ID, App ID, Secret** for next tasks.

New-OrganizationRelationship -Name "NSI-OrgRel" `

Below is a “peer-review” of the backlog from the viewpoint of a Microsoft 365 (Office 365) migration architect who has run cross-tenant projects since 2013. ---

## 1   Strategy-level observations

| Area | Comments and recommendations |
|------|------------------------------|
| **Governance and change management** | • Add a short communication plan: weekly “what’s coming” e-mails, a cut-over FAQ, and a one-page cheat-sheet for signing in to the new tenant. Low effort, high reduction in help-desk noise. <br>• Create a single Microsoft Teams “War-Room” channel with @mentions for blockers. |
| **Pilot before production** | Even though the scope is small, run the migration steps for **one** SharePoint site and **one** mailbox first. This surfaces throttling limits, PowerShell MFA prompts, and naming collisions. |
| **Rollback / safety** | Keep NSI sites read-write for *30 days* after migration (even if you later plan to lock them) so you can delta-copy any missed files. |
| **Timeline realism** | Manual PST export/import is slow: ~5–8 GB per hour on a well-connected PC. If the six NBI-only mailboxes are > 20 GB each, schedule two evenings or one weekend just for mail. |

---

## 2   Detailed technical notes

### 2.1   SharePoint / OneDrive

| Topic | Review and suggestions |
|-------|-----------------------|
| **SPMT input** | Feeding the SharePoint Migration Tool one library at a time is fine, but you can bulk-load with a JSON task file (Microsoft’s “Bulk Migration” format). Saves click-time if you have more than ~10 libraries. |
| **Version limits** | The plan keeps “last 500 versions”. If any libraries use “Major + Minor” drafts, that truncation might drop history. Confirm with site owners. |
| **Hidden artefacts** | SPMT does not copy: • Site Collection App Catalog packages • Classic publishing pages • SharePoint Framework Application Customizers. If any NSI site has these, flag them for manual work. |
| **Teams private channels** | Private-channel file areas live in *separate* SharePoint sites (‐private suffix). Add them to the worksheet if the business uses Teams heavily. |
| **Power Automate / Power Apps** | Neither SPMT nor PnP templates migrate flows or canvas apps. Create a line-item: “Re-create Power Automate flows and Power Apps pointing to new NBI lists” if they exist. |

### 2.2   Exchange Online

| Topic | Review and suggestions |
|-------|-----------------------|
| **Export / import method** | Outlook PST export/import works but: • Requires Outlook licensed on an admin PC. • Losses: category colours, Inbox rules, shared mailbox permissions. <br>Alternative still free: **Content search eDiscovery export** from the Microsoft Purview portal. Creates a single .pst faster and does not need Outlook. |
| **AutoComplete (nickname) cache** | Users moving to a new profile lose AutoComplete history. Warn them and provide the path to copy .nk2 (for Outlook 2016/2019) if that is critical. |
| **Calendar sharing and delegates** | Delegations do not survive PST import. Capture “Send As / Full Access / Editor” relationships before you export so you can recreate them in NBI. |
| **Mobile devices** | If mobiles use the built-in iOS or Android mail app via Exchange ActiveSync, document that they must delete the NSI profile and add the NBI profile; otherwise devices may keep hitting the old tenant. |

### 2.3   Identity and security

| Topic | Review and suggestions |
|-------|-----------------------|
| **Password resets on first sign-in** | When you create the 12 users you chose *auto-generated passwords*. Set “Require change on next sign-in” to **Yes** so you are not tracking long-term secrets in a vault. |
| **Conditional Access** | If NSI uses Conditional Access policies (for example, block sign-in outside specific countries) copy equivalent rules to NBI before user cut-over. |
| **Guest accounts** | Any external guest users in NSI SharePoint will need new guest invites in NBI if they still collaborate with NBI-owned sites. |

---

## 3   Ordering improvements

1. **Create NBI domain first** (current plan creates users first). If you add the NBI-Consulting.com domain upfront, you can assign their final address on Day 1 instead of renaming later.
2. **MFA enablement**: enable *after* the first sign-in to reduce the “immediate MFA registration” screen during bulk account creation.
3. **PnP export**: do it *after* SPMT if you want modern pages updated by the site owners to come across last-minute; otherwise branding arrives but later site-changes do not.

---

## 4   What is not covered (but often requested)

| Omitted element | Typical mitigation (still zero-cost) |
|-----------------|---------------------------------------|
| Microsoft Teams chat history | Only third-party tools move chats. Instead, keep NSI Teams tenant for chat history. |
| Planner plans, To Do tasks | Export via PowerShell Graph to JSON then re-create; labour-intensive, consider re-planning instead. |
| Viva Insights personal data | Does not migrate; it rebuilds automatically in the new tenant after activity. |

---

## 5   “No licence spend” side-effects

* **Manual labour**: Export/import PST and GUI SPMT may take more admin hours than a Cross-Tenant Mailbox Move or a third-party SharePoint tool. Track that in the project budget even if licence cost is zero.
* **Throttling**: Free SPMT jobs are limited by Microsoft service throttles (especially during business hours). Run high-volume tasks evenings or weekends to avoid failures.

### Microsoft Purview in plain English

**Microsoft Purview** is Microsoft’s umbrella brand for all of its **data-governance, compliance, risk-management, and discovery products**—spanning Microsoft 365, Azure, on-premises systems, and even other clouds (AWS, Google Cloud). Think of it as everything you use to *find, classify, protect, audit, search, and govern* data, wrapped into one named suite.

---

#### Two big “families” under the Purview name

| Purview area | What it does | Typical use cases |
|--------------|--------------|-------------------|
| **Microsoft Purview Compliance Portal**<br>(formerly the *Microsoft 365 Compliance Center*) | • Runs inside the Microsoft 365 admin experience.<br>• Covers Information Protection labels, Data Loss Prevention, eDiscovery (Standard & Premium), Content Search, Insider-Risk Management, Audit, Communication Compliance, Records Management, Compliance Manager score, Privacy Risk Assessments, etc. | • Search and export a custodian’s mailbox (our migration plan’s *Content search > Export to PST* step).<br>• Apply retention labels to SharePoint docs.<br>• Show audit trail of “who accessed this file”. |
| **Microsoft Purview Data Governance**<br>(evolved from *Azure Purview*) | • Cloud service that scans data sources (Azure Data Lake, SQL, AWS S3, on-prem file shares).<br>• Builds a data map, auto-classifies sensitive data, tracks lineage, lets you publish a Data Catalog. | • Enterprise needs a single inventory of where customer-data fields live.<br>• BI team wants to see lineage of a Power BI dataset back to raw tables. |

*Branding note*: In late 2022 Microsoft re-branded **Azure Purview + Microsoft 365 Compliance** under the single “Microsoft Purview” banner. So you’ll see both portals, but they share the same label framework and reporting backbone.

---

#### Why Purview mattered in your migration plan

* **eDiscovery / Content search** – We chose it to export each NSI mailbox to a .pst because it is quicker, scriptable, and licence-included for E3/Business Standard tenants versus using Outlook desktop exports.
* **Audit & hold** – If NSI has retention policies, Purview keeps inactive mailboxes or deleted SharePoint sites on legal hold—important if you need historical evidence post-migration.
* **Data-classification carry-over** – Any Microsoft Purview “Sensitivity Labels” applied to files or mail stay attached after we copy them to the NBI tenant (because they are embedded in the file).

---

#### Licensing at a glance (2025)

| Feature | Included in | Extra cost? |
|---------|-------------|-------------|
| Content Search, Core Audit, Sensitivity Labels, Retention Policies | Microsoft 365 Business Standard, E3 | No |
| eDiscovery Premium (custodian holds, advanced review), Audit Premium (1-year log), Insider-Risk Mgmt | Microsoft 365 E5 or “E5 Compliance” add-on | Yes |
| Purview Data Governance (Data Map, Catalog) | Pay-as-you-go Azure service | Consumption-based |

---

##### TL;DR – Microsoft Purview  
A consolidated brand for **everything Microsoft sells to help you discover, classify, protect, govern, audit, and legally review data**, whether that data lives in Microsoft 365, Azure, on-prem, or another cloud. In our migration playbook we used the *Purview Compliance Portal’s Content search* so we could export mailboxes without paying for third-party tools or running Outlook drag-and-drops.
