# **STATEMENT OF WORK (SOW)**
**Development of a Data Library Tool for Game Telemetry Management**

## **1. Project Overview**
### **Objective**
The purpose of this project is to design and develop a **frontend tool** that streamlines the **definition, management, and versioning of game telemetry events**. This tool will enable **data analysts, engineers, and game team members** to collaborate efficiently, ensuring standardized event creation, structured data management, and seamless version control using **Git-based storage**.

The project will **enhance traceability, maintainability, and usability** by enforcing **naming conventions**, **version tracking**, and **review workflows** while providing **a user-friendly UI** for configuring event telemetry.

---

## **2. Key Stakeholders**
- **Data Architects / Engineers** – Define and manage event components based on input from analysts and game teams.
- **Data Analysts** – Assemble predefined components into events, request changes/additions.
- **Game Team Members** – Access and review proposed event logic for alignment with game design and live service strategies.

---

## **3. Scope of Work**
The development of the **Data Library Tool** will focus on two primary modules:

### **3.1 Event Component Definition & Management**
**Goal:** Ensure a standardized structure for game telemetry events using predefined building blocks.

#### **Requirements:**
- A **structured and user-friendly UI** to define event components.
- Enforce **standardized naming conventions** for consistency.
- Version control for components with clear status indicators:  
  _(e.g., Draft, Prototype, Ready, Pending, Approved)._
- Ability to **compare versions of components** to track modifications.
- Store event components as **code in Git** for traceability and consistency.
- Review workflow to **approve/reject changes** before they become active.

### **3.2 Event Configuration**
**Goal:** Allow analysts to efficiently define **telemetry events per project** while ensuring completeness and reviewability.

#### **Requirements:**
- UI to **browse and select predefined event components**.
- Ability to **assemble an event configuration** using available components.
- Define **event triggering logic** (context of when/how the event fires).
- Submit events for **review and approval** before finalization.
- Versioning of events, stored as **code in Git** for consistency.
- Comparison features:
  - **Different versions of an event** _(to track modifications over time)._
  - **Same event across multiple game projects** _(to ensure consistency)._
- Indicator for **missing/incomplete parameters** required for event definition.
- Request mechanisms:
  - **Update existing components** (e.g., adding new values).
  - **Create new components** (if a required one does not exist).
- **Notification system** to alert users of pending requests and track resolution.
- Stretch goal: **Email/Slack notifications** for updates on approvals or requests.

# **STATEMENT OF WORK (SOW)**  
**Development of a Product Plan for the Data Library Tool**

### **Objective**  
This project aims to develop a **comprehensive product plan** for a Data Library Tool that will streamline the **definition, management, and versioning of game telemetry events**. The product plan will provide a **clear roadmap, functional requirements, design specifications, and feasibility assessments** to guide the full development of the tool.  

The scope of this SOW **does not include** actual tool development but focuses on the **research, definition, and planning phases** necessary to ensure a well-structured and effective implementation strategy.

---

## **2. Scope of Work**  
The project will be conducted in **four key phases**:  

1. **Discovery & Requirements Gathering** – Investigating needs, analyzing current workflows, and documenting key use cases.  
2. **Feasibility & Technology Assessment** – Evaluating technical constraints, dependencies, and available integrations.  
3. **Design & Specification** – Defining architecture, UI/UX framework, and functional requirements.  
4. **Product Roadmap & Planning** – Creating a phased development roadmap, risk assessment, and milestone planning.  

Each phase will involve **collaboration with stakeholders** to ensure alignment with business and technical needs.

---

- **Product Owner** – Ensures alignment with business objectives.  
- **Data Architects / Engineers** – Provide technical insights on data infrastructure and telemetry event structures.

- **Game Team Members** – Review how telemetry events will integrate with game development and live operations.  
- **Project Manager** – Oversees planning and coordination.

## **4. Deliverables**  

### **Phase 1: Discovery & Requirements Gathering**  
**Goal:** Define the scope, business requirements, and key use cases for the Data Library Tool.  
- **Stakeholder Interviews** – Gather insights from data architects, analysts, and game developers.

- **Use Case Documentation** – Define primary functionalities needed for the tool.

✅ **Requirements Document** – Captures all business needs, user workflows, and functional requirements.

---

### **Phase 2: Feasibility & Technology Assessment**  
**Goal:** Evaluate technical constraints, data storage options, and integration requirements.  
- **Review of Existing Data Infrastructure** – Assess how the tool integrates with Git, databases, and telemetry pipelines.  
- **Version Control Strategy** – Define best practices for Git-based storage and retrieval.  
- **Technology Stack Assessment** – Compare options for frontend, backend, and hosting solutions.  
- **Security & Compliance Evaluation** – Identify data security requirements and regulatory considerations.  

**Deliverable:**  
✅ **Feasibility Report** – Outlines technical challenges, constraints, and recommended technology choices.

---

### **Phase 3: Design & Specification**  
**Goal:** Define the detailed **architecture, user experience, and functional design** of the tool.  
- **UI/UX Wireframing** – Design user interface layouts and workflows.  
- **Data Model Design** – Define data structure for storing event components and metadata.  
- **Versioning & Approval Workflow** – Define how event modifications will be tracked, reviewed, and approved.  
- **Integration Framework** – Specify API endpoints for game telemetry ingestion and retrieval.  
- **User Roles & Permissions** – Define access control for data analysts, engineers, and reviewers.  

**Deliverable:**  
✅ **Design Specification Document** – Includes UI wireframes, architecture diagrams, and detailed feature descriptions.

---

### **Phase 4: Product Roadmap & Planning**  
**Goal:** Develop a phased rollout plan with milestones, dependencies, and risk management strategies.  
- **Development Phases & Timeline** – Define iterative releases and MVP scope.  
- **Dependency Mapping** – Identify dependencies on internal systems, APIs, and external integrations.  
- **Risk Assessment & Mitigation Plan** – Address potential challenges and backup strategies.  
- **Resourcing & Budgeting Estimate** – Outline resource allocation for development.  

**Deliverable:**  
✅ **Product Roadmap Document** – Includes feature milestones, risks, and development timeline.
