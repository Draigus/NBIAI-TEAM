## Cross-cutting platform requirements
**Description**  
Foundational capabilities shared by the suite: identity, workspaces, catalogue, evidence and definitions, SLOs, observability.

**Value**  
Speeds delivery. Reduces duplication. Enables trust and governance.

**Core features and requirements**  
1) Identity and access  
- What: SSO with Google and Microsoft; roles Admin, Editor, Viewer; workspace membership; API keys per workspace.  
- Why: secure sharing and least-privilege by default.  
- Technical: OAuth 2.0, OIDC; JWT with rotation; workspace-scoped RBAC; audit log of auth events.  
- Requirement: Users can sign in with SSO, belong to 1+ workspaces, and receive role-based rights at object and field level.  
- Acceptance: Create user via SSO, invite to workspace, verify role changes reflected within 60 seconds in UI and API.

2) Workspace objects  
- What: Titles, Companies, Platforms, Stores, Regions, Genres, Calendars.  
- Why: consistent joins across modules.  
- Technical: Normalised Postgres tables; global IDs; soft deletes; versioned dictionaries; referential integrity.  
- Requirement: All tools read from the same objects; dictionary updates are versioned and backward compatible.

3) Evidence and definitions  
- What: hover cards showing metric definitions, last refresh, and provenance; parity mode that normalises external vendor definitions.  
- Why: reduce metric disputes and speed decisions.  
- Technical: Definition registry; provenance table keyed to tile and metric; UI hovers; Estimated flag when stale.  
- Requirement: Every chart and metric exposes definition and provenance, with a visible Updated timestamp.

4) Performance SLOs  
- What: Tiles Ōēż 2 s on seeded data; filter changes Ōēż 3 s; exports Ōēż 120 s; alert latency Ōēż 5 min from rule trigger.  
- Why: investor-safe demos and operator-grade UX.  
- Technical: server-side caching with per-workspace cache keys; async export workers; event queue for alerts; SLO dashboards.  
- Requirement: SLOs met in test and logged in Grafana. Failing SLOs raise an Ops ticket automatically.

5) Observability and audit  
- What: application logs, metric time series, traces, and full admin audit log; data lineage view from tile to source.  
- Why: debug speed, trust, compliance.  
- Technical: OpenTelemetry; Prometheus; Grafana; immutable audit table; lineage stored in a graph table.  
- Requirement: For any metric, show lineage and last refresh. For any role change or export, show who, what, when.

2) Audit and retention  
- What: user actions, admin changes, and data refresh logs; retention policy.  
- Why: accountability.  
- Technical: append-only audit table; exportable CSV; purge jobs.  
- Requirement: All admin actions are searchable by user and time.

**Why build**  
Necessary for larger partners and investors.

**Technical needs**  
Central policy service, logging, retention schedulers.

**Acceptance tests**  
Attempted blocked export raises a clear error; audit shows who changed roles and when.

# Engineering foundations

**Frontend**  
React with Next.js, TypeScript, Tailwind. State: React Query and Zustand. Charting: Recharts or ECharts. Accessibility WCAG AA.

**Backend**  
FastAPI or Node with NestJS. REST plus optional GraphQL. Background jobs with Celery or BullMQ. Cache: Redis. Search: Postgres trigram or OpenSearch if required.

**Data**  
Postgres as system of record. Columnar acceleration with DuckDB or ClickHouse. Dagster for orchestration. dbt for transforms. Object storage for snapshots and exports.

**ML**  
Time series models for Foresight, embeddings for Sentiment, light ranking for The Sage. Model registry, feature store, scheduled retrains.

**Security**  
OIDC SSO, RBAC, secrets in a vault, signed URLs, encryption at rest and in transit, IP allow lists, export watermarking.

**Ops**  
Kubernetes deployment, Helm charts, autoscaling, Prometheus and Grafana, OpenTelemetry traces, on-call runbook.
