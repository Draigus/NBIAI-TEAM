By adopting a similar event scheduling mechanism, the Sarge forecast can precisely model the impact of scheduled content releases (new game modes, major updates), marketing campaigns, or special events on user numbers and revenue. For example, instead of a flat 3% bump for “holiday events” across the board, we could have an Events table where each event (with a date or week number) can input a specific uplift (e.g. a content drop in Week 20 brings +50k new users and +10% spend for 2 weeks). This granular control will make the forecast more realistic and scenario-driven, as one can turn events on/off to see their contributions.

Granularity & Segmentation: The Apex model’s breakdown by platform may not directly apply to Sarge (since Sarge is a single-platform mobile game), but it underscores the importance of segmentation. The idea is to ensure that the model can capture differences in user behavior and monetization for different groups, rather than treating the entire user base homogenously. Segmentation improves accuracy and insight (e.g. identifying that early adopters might monetize differently than later adopters).

Design for an Enhanced Sarge Universe Forecast Simulator
Using the insights from both models, we will now outline the structure and features of the new Excel-based forecast simulator for Sarge Universe. This design focuses on clarity, flexibility, and completeness, treating the forecast model as a product deliverable for stakeholders. The model will be organized into several sheets, each with a clear purpose, and will include mechanisms for scenario selection, control toggles for key metrics, and visually appealing outputs. All formulas will be carefully implemented and tested for accuracy. An Instructions tab will be provided to guide users through the model. Below is the breakdown of the proposed model components:
1. Control Panel & Assumptions
This sheet will be the front page of the model where a user can input or adjust all key assumptions in one place. It will include:
•	Scenario Selection: A dropdown or toggle cell to switch between Low, Medium, High scenarios (e.g., cell B2 might have a Data Validation list of "Low/Med/High"). This selection will drive which set of assumptions apply (possibly using formulas like =IF(Scenario="Low", LowCaseValue, IF(Scenario="High", HighCaseValue, BaseValue)) or by referencing a scenario table). The Apex model’s approach of having modifier percentages for each scenario can be employed: for instance, the base case assumptions are listed in one column, and adjacent columns provide % adjustments for Low/High. When a scenario is chosen, those adjustments are applied across the model. This way, the client can easily see how metrics change under different conditions without editing individual cells.
•	Key Metrics & Drivers: A structured table of all main input drivers, grouped by category for readability. For example:
•	User Acquisition Inputs: initial launch installs (or weekly new users at launch), growth/decay rate of new users post-launch, any plateau value for steady-state acquisitions, referral or virality factor (if any), etc. If multiple acquisition channels (organic vs paid) are considered, each can have its own line. For each input, provide a base value, and low/high adjustments if applicable. (E.g., Base first-month installs = 12,000,000; Low scenario maybe 20% lower, High 20% higher, or based on Apex we saw Low Modifier 0, High -10% etc., we will tailor the percentages appropriately).
•	Retention & Return Rates: initial retention percentages (D7, D30, or weekly retention rate) and how it changes over time if applicable, as well as any percentage of users returning in later weeks. For example, Starting retention might be 25% (meaning maybe 25% of Week 1 new users are still active by Week 2), and a decay factor could reduce retention as weeks progress. Return rate could be, say, 10-15% of dormant users returning when new content appears. These can be scenario-toggled (perhaps optimistic scenario assumes slightly better retention).
•	Engagement Metrics: average playtime or sessions per user if needed for any calculations, but more importantly conversion of actives to payers (e.g. Conversion Rate to Payer in month 1, etc.) and Avg Revenue per Paying User. We will include assumptions like Battle Pass uptake rate (e.g. X% of total actives buy the pass) and Avg spend on IAP per payer per month (if not directly computing from more granular price tiers).
•	Monetization Price Points: key price points such as Battle Pass price (e.g. $4.99/month), higher-tier pass price if applicable ($19.99 for “Commander” tier), and average price of lootbox or IAP purchases. If the model uses ARPDAU (average revenue per daily active user) or ARPPU, those values or their components would be listed here for transparency. For instance, ARPDAU could be an output of other assumptions but we might list an expected ARPDAU to sanity-check the model’s results.
•	In-Game Economy Rates: assumptions such as what fraction of purchases are of the premium currency (credits) vs direct items, the exchange rate of real money to credits (e.g. $1 = 100 credits?), and how much of earned credits are spent (sink rate). Also, if any sinks are particularly large (e.g. a certain feature like upgrades consumes a lot of credits), we can set a parameter for that if it affects monetization indirectly (though initially, sinks mostly matter for economy balance, not revenue).
•	Growth Events: rather than burying event impacts in formulas, we can list in the control panel some high-level modifiers for known big events. For instance, “Major Update User Boost = +20% new users during that week” or “Holiday Spending Lift = +10% revenue in December”. These can then feed into the events sheet or directly into calculations. Making them visible in control panel allows quick tweaking of how powerful we expect events to be in general scenarios.
Each assumption will be clearly labeled and, where possible, accompanied by a brief note or unit (e.g., percentages, absolute values, currency). We will color-code input cells (commonly, light yellow or another color for inputs, and keep formula cells in a different color or locked). The control panel essentially acts as the “dashboard of inputs” for the entire model, so the client can review or modify assumptions without digging into the calculation sheets.
 
Additionally, having all key inputs on one sheet makes scenario comparison easier – we could even display side-by-side columns for Low/Base/High for some inputs so a user can visually compare the scenarios’ assumptions in one view.
2. User Acquisition & Growth Model (Weekly Detail)
This section of the model will calculate the number of new users joining the game each week and project the growth trajectory of the user base. It will likely span one sheet (e.g., called “Acquisition & Users” or simply “WeeklyModel”) that lists weeks sequentially (from Week 1 of launch onward).
 
Key elements of this sheet:
•	Timeline Structure: We will list weeks (and months) as rows, with a starting date for launch. For example, Week 1 could correspond to the launch week (with an actual date or a label “Week 1”), and we will number weeks up to the desired horizon (e.g., 2-3 years of weekly data, which is ~156 weeks for 3 years, or potentially 260 weeks for 5 years if we go that far). We can also have a corresponding Month column to aggregate or mark the month each week falls into, which will help in building monthly summaries.
•	New Users Calculation: For each week, calculate New Users (NUU). The model from Sarge provided an initial monthly install number; we can distribute that into weekly values for the first month. The Apex model used a decaying function (Start value and a decay factor per week). We can adopt a similar approach:
•	We set Week 1 new users as a certain number (e.g., 3 million if 12M in first month spread across 4 weeks, or we can front-load if we expect a launch spike).
•	Each subsequent week’s new users might equal the prior week’s new users multiplied by a decay factor. For example, if “Acquisition Decay” is -1.5 (as seen in Apex for PC, meaning maybe a 15% weekly drop if interpreted as a multiplier like 0.85 each week), we apply that to simulate declining new installs over time as the initial hype tapers.
•	We may incorporate seasonal bumps: If the Season sheet or an Events table dictates a bump at certain points, the new user count for those weeks will include the extra users. E.g., if in Season 2 Week 1 there’s a content launch, we might add a fixed number or percentage increase to that week’s new users.
•	If we consider multiple acquisition channels (like organic vs paid), we might have columns for each and sum to total new users. In early stages, user acquisition could heavily rely on marketing (paid UA), which might be capped by budget – however, since we’re not given marketing spend details here, we may keep it simpler and treat new users as an aggregate with an implicit assumption of marketing effort included.
•	Returning Users: If the Sarge model’s “Return” rate (14% in control panel) is meant for users who had churned and come back with new content, we will incorporate that logic. One way is: each time a new Season or major update starts, take a percentage of the “cumulative churned users” and add them as “returning users” in that week. Alternatively, have a modest ongoing returning user flow each week (e.g., each week a certain % of all previously lost users return, which creates a long tail of re-engagement). We’ll likely implement it by marking specific event weeks where return spikes happen (e.g., if Return = 14%, perhaps 14% of total lost users come back during a big update week). This can be part of the Events effects rather than a constant each week, since typically players return when there’s new content.
•	Active User Calculation: While new users tell us how many are coming in, we need to calculate how many active users exist in each week. This involves retention:
•	We will maintain columns for Cohort Retention or a simpler aggregate retention. A detailed way is to simulate cohorts: each week’s new users are a cohort, and we apply a retention curve to see how many of that cohort are still active in subsequent weeks. However, a full cohort model can be heavy in Excel (but feasible over a few years weekly).
•	A simpler approach (if cohorts are not absolutely required) is to use a rolling retention multiplier: e.g., “active users this week = active users last week minus some who churn plus any new and returning users”. But to stay accurate, a cohort model is better because churn rates change as cohorts age.
•	Given the importance of retention in gaming, we may do a lightweight cohort: e.g., have a fixed D7, D30 retention and assume an attrition curve beyond that (like a weekly survival rate).
•	Perhaps use the known retention: If starting retention (maybe D7 or first week retention) is 25%, that implies out of Week1 new users, 25% continue to week2. Then perhaps retention from week to week might stabilize at some percentage.
•	We can use an exponential decay or a fixed weekly retention percentage (e.g., assume each active user has X% chance to stay to next week). For initial simplicity: assume a weekly retention rate like 80% per week for the average user (just as an example), which would yield a certain decay over time. This can be adjusted for scenario (High scenario might have higher retention).
•	The Retention sheet from Apex likely had these rates, and we can incorporate such a table (maybe in Key Parameters or a hidden calc sheet). For instance, define retention % for week1->2, week2->3, etc., or monthly retention per month of life.
•	Using those, calculate for each cohort of new users how many remain active each subsequent week and sum across all cohorts to get total active users in a given week.
•	The result of this will be a column for Weekly Active Users (WAU) or since we are doing weekly, WAU might simply be the active users that week. If we want DAU, we could assume a certain % of WAU log in daily, or use WAU as a proxy (or multiply WAU by frequency factor to get total daily sessions if needed for ARPDAU).
•	We will also calculate Monthly Active Users (MAU), likely on the monthly summary sheet by taking the unique users active in a month. But since we have weekly, an approximation is to take the max of 4-week active or something. We might just report WAU or an average DAU for simplicity, depending on how monetization is modeled (often ARPDAU is easier if DAU is known).
•	Link to Monetization: The output of this sheet – particularly the number of active users each week (and possibly the number of new payers) – will feed into the monetization sheets. For example, if we know how many active users are in week 10, and what percentage of them make a purchase that week, we can compute that week’s revenue. Alternatively, if we compute an ARPDAU, we multiply by DAU. We will structure it so that either approach can be taken. Likely, we’ll have a column for Payer Conversion (number of paying users in that week) equal to active users * conversion rate (which might diminish over time or increase as the game matures and hardcore players remain). We can draw conversion rates from assumptions (e.g., 5% of actives pay in the first month, perhaps trending upward later).
In summary, the Acquisition & Growth section provides the core user funnel over time: new users -> active users -> paying users. It will be the engine that drives the rest of the forecast, and it will be sensitive to scenario inputs (higher growth in High scenario, lower in Low, etc.).
3. Monetization Modules (Revenue Streams)
Instead of one giant sheet, we will create separate sheets (or sections on one sheet) for each major revenue stream, similar to how both existing models do. In summary, the Sarge Universe model provides a solid foundation with detailed game-specific components (engagement, multiple monetization categories, in-game currency tracking, seasonal effects). This granular control means the forecast isn’t just a blunt trend line – it accounts for spikes due to specific drivers (feature releases, marketing beats, partnerships). For Sarge Universe, which will have periodic events (tournaments, new content, etc.), such a mechanism could greatly enhance realism in the forecast.

This segmented approach ensures that the model can capture differences in user behavior or revenue per platform (console players might monetize differently than PC players, for example). While Sarge Universe is primarily a Telegram/mobile game (single platform), the concept of segmentation could still apply to different user groups or release channels (e.g. distinguishing organic users vs referred users, or geographic segments), if needed. Given Apex is a code name (possibly for a project or just an example), the battle pass sheet likely calculates how many players purchase the pass each season, considering conversion rates of active users to battle pass buyers and perhaps a price point. Key Differences and Insights
From the above comparison, several key differences emerge, highlighting opportunities to improve the Sarge model:
1.	Scenario Analysis: The Apex model’s ability to switch between Low, Medium, High cases is a clear advantage. The Sarge model currently lacks this (only one set of inputs). Incorporating a scenario framework in Sarge’s forecast will allow the team to plan for uncertainty by examining best-case, base-case, and worst-case outcomes. This means adding Low/Med/High variants for critical metrics like user acquisition, retention, ARPU, conversion rates, etc., and an easy way to toggle which set is active (e.g. a dropdown or a scenario input cell that formulas reference). This is standard in robust financial models and will make the Sarge forecast far more flexible for stakeholder presentations.
2.	Event/Feature Impact Modeling: Sarge’s model includes a simple holiday event bump and seasonal concept, but Apex goes further with a dedicated feature timeline. By adopting a similar event scheduling mechanism, the Sarge forecast can precisely model the impact of scheduled content releases (new game modes, major updates), marketing campaigns, or special events on user numbers and revenue. For example, instead of a flat 3% bump for “holiday events” across the board, we could have an Events table where each event (with a date or week number) can input a specific uplift (e.g. a content drop in Week 20 brings +50k new users and +10% spend for 2 weeks). This granular control will make the forecast more realistic and scenario-driven, as one can turn events on/off to see their contributions.
3.	Granularity & Segmentation: The Apex model’s breakdown by platform may not directly apply to Sarge (since Sarge is a single-platform mobile game), but it underscores the importance of segmentation. We can consider segmenting Sarge’s users by acquisition channel or user type (for instance, organic vs paid acquisition, or perhaps new users vs returning veterans). Another useful segmentation might be by player cohorts (e.g. new players acquired in each month or season, and track their behavior separately). The idea is to ensure that the model can capture differences in user behavior and monetization for different groups, rather than treating the entire user base homogenously. Segmentation improves accuracy and insight (e.g. identifying that early adopters might monetize differently than later adopters).
4.	Monetization Detail: Both models have a rich breakdown of revenue streams. Sarge’s model already shines in differentiating revenue sources (battle pass, lootbox, etc.), which is great. Apex adds possibly a licensing/partners revenue component; if Sarge has any external revenue (perhaps NFT sales were considered originally, or future licensing/IP deals), those could be an optional component. However, since Sarge is refocusing on in-game currency only (no NFTs/tokens for now), we will likely keep monetization streams to in-app/content sales only. One area to borrow from Apex is how each category’s outputs roll up cleanly into a financial summary. We will ensure the new design has a clear summary tab where all these streams sum up to total revenue by month/season, rather than having to pull numbers from multiple sheets.
5.	In-Game Economy (Sinks & Sources): Sarge’s model uniquely tracks currency flow, which the Apex financial model doesn’t explicitly show (Apex’s “Coin Calculator” prototype suggests they were toying with a similar idea). We should carry this forward, making it a distinctive feature of Sarge’s forecast simulator. This means not only forecasting revenue (money in) but also tracking the currency given to players (from rewards, etc.) versus currency spent. Maintaining a balance of the currency economy can highlight if the game might face currency inflation or if players will run out of currency (which can inform tuning of prices or rewards). It’s a level of detail that impresses clients because it shows a holistic view of the game’s economy, not just top-line revenue.
6.	Outputs & Visualization: The Apex model is better geared for presenting results (with its summary sheets and likely charts). The Sarge model would benefit from a dedicated Dashboard/Graphs sheet. By comparing, we see that having weekly and monthly summaries and key charts (e.g. cumulative revenue, monthly active users trend, revenue by category, etc.) makes a forecast model far more digestible. Additionally, the Apex model’s use of actuals indicates a practice of comparing forecast vs real data; while that might be beyond scope for a pre-launch forecast, building the Sarge model in a way that actuals can be plugged in later (to re-forecast or check assumptions) would future-proof it.
7.	Excluded Elements: We note that any references to blockchain tokens or ad revenue present in prior Sarge documents will be excluded in the new model. The focus is on core game revenue streams (in-app purchases, possibly subscription if battle pass is considered one) and in-game currency dynamics. This simplifies the model and aligns it more with Apex’s traditional game revenue approach. (Apex model’s context suggests it’s a premium/Free-to-Play console/PC game with no ads or tokens, matching what we want for Sarge).
In summary, the Apex model contributes valuable features such as scenario toggling, event-driven forecasting, and polished summary outputs, while the Sarge model brings game-specific depth (like currency economy and detailed IAP categories). The goal is to merge these strengths into the ultimate Sarge Universe forecast simulator.
Design for an Enhanced Sarge Universe Forecast Simulator
Using the insights from both models, we will now outline the structure and features of the new Excel-based forecast simulator for Sarge Universe. This design focuses on clarity, flexibility, and completeness, treating the forecast model as a product deliverable for stakeholders. The model will be organized into several sheets, each with a clear purpose, and will include mechanisms for scenario selection, control toggles for key metrics, and visually appealing outputs. All formulas will be carefully implemented and tested for accuracy. An Instructions tab will be provided to guide users through the model. Below is the breakdown of the proposed model components:
1. Control Panel & Assumptions
This sheet will be the front page of the model where a user can input or adjust all key assumptions in one place. It will include:
•	Scenario Selection: A dropdown or toggle cell to switch between Low, Medium, High scenarios (e.g., cell B2 might have a Data Validation list of "Low/Med/High"). This selection will drive which set of assumptions apply (possibly using formulas like =IF(Scenario="Low", LowCaseValue, IF(Scenario="High", HighCaseValue, BaseValue)) or by referencing a scenario table). The Apex model’s approach of having modifier percentages for each scenario can be employed: for instance, the base case assumptions are listed in one column, and adjacent columns provide % adjustments for Low/High. When a scenario is chosen, those adjustments are applied across the model. This way, the client can easily see how metrics change under different conditions without editing individual cells.
•	Key Metrics & Drivers: A structured table of all main input drivers, grouped by category for readability. For example:
•	User Acquisition Inputs: initial launch installs (or weekly new users at launch), growth/decay rate of new users post-launch, any plateau value for steady-state acquisitions, referral or virality factor (if any), etc. If multiple acquisition channels (organic vs paid) are considered, each can have its own line. For each input, provide a base value, and low/high adjustments if applicable. (E.g., Base first-month installs = 12,000,000; Low scenario maybe 20% lower, High 20% higher, or based on Apex we saw Low Modifier 0, High -10% etc., we will tailor the percentages appropriately).
•	Retention & Return Rates: initial retention percentages (D7, D30, or weekly retention rate) and how it changes over time if applicable, as well as any percentage of users returning in later weeks. For example, Starting retention might be 25% (meaning maybe 25% of Week 1 new users are still active by Week 2), and a decay factor could reduce retention as weeks progress. Return rate could be, say, 10-15% of dormant users returning when new content appears. These can be scenario-toggled (perhaps optimistic scenario assumes slightly better retention).
•	Engagement Metrics: average playtime or sessions per user if needed for any calculations, but more importantly conversion of actives to payers (e.g. Conversion Rate to Payer in month 1, etc.) and Avg Revenue per Paying User. We will include assumptions like Battle Pass uptake rate (e.g. X% of total actives buy the pass) and Avg spend on IAP per payer per month (if not directly computing from more granular price tiers).
•	Monetization Price Points: key price points such as Battle Pass price (e.g. $4.99/month), higher-tier pass price if applicable ($19.99 for “Commander” tier), and average price of lootbox or IAP purchases. If the model uses ARPDAU (average revenue per daily active user) or ARPPU, those values or their components would be listed here for transparency. For instance, ARPDAU could be an output of other assumptions but we might list an expected ARPDAU to sanity-check the model’s results.
•	In-Game Economy Rates: assumptions such as what fraction of purchases are of the premium currency (credits) vs direct items, the exchange rate of real money to credits (e.g. $1 = 100 credits?), and how much of earned credits are spent (sink rate). Also, if any sinks are particularly large (e.g. a certain feature like upgrades consumes a lot of credits), we can set a parameter for that if it affects monetization indirectly (though initially, sinks mostly matter for economy balance, not revenue).
•	Growth Events: rather than burying event impacts in formulas, we can list in the control panel some high-level modifiers for known big events. For instance, “Major Update User Boost = +20% new users during that week” or “Holiday Spending Lift = +10% revenue in December”. These can then feed into the events sheet or directly into calculations. Making them visible in control panel allows quick tweaking of how powerful we expect events to be in general scenarios.
Each assumption will be clearly labeled and, where possible, accompanied by a brief note or unit (e.g., percentages, absolute values, currency). We will color-code input cells (commonly, light yellow or another color for inputs, and keep formula cells in a different color or locked). The control panel essentially acts as the “dashboard of inputs” for the entire model, so the client can review or modify assumptions without digging into the calculation sheets.
 
Additionally, having all key inputs on one sheet makes scenario comparison easier – we could even display side-by-side columns for Low/Base/High for some inputs so a user can visually compare the scenarios’ assumptions in one view.
2. User Acquisition & Growth Model (Weekly Detail)
This section of the model will calculate the number of new users joining the game each week and project the growth trajectory of the user base. It will likely span one sheet (e.g., called “Acquisition & Users” or simply “WeeklyModel”) that lists weeks sequentially (from Week 1 of launch onward).
 
Key elements of this sheet:
•	Timeline Structure: We will list weeks (and months) as rows, with a starting date for launch. For example, Week 1 could correspond to the launch week (with an actual date or a label “Week 1”), and we will number weeks up to the desired horizon (e.g., 2-3 years of weekly data, which is ~156 weeks for 3 years, or potentially 260 weeks for 5 years if we go that far). We can also have a corresponding Month column to aggregate or mark the month each week falls into, which will help in building monthly summaries.
•	New Users Calculation: For each week, calculate New Users (NUU). The model from Sarge provided an initial monthly install number; we can distribute that into weekly values for the first month. The Apex model used a decaying function (Start value and a decay factor per week). We can adopt a similar approach:
•	We set Week 1 new users as a certain number (e.g., 3 million if 12M in first month spread across 4 weeks, or we can front-load if we expect a launch spike).
•	Each subsequent week’s new users might equal the prior week’s new users multiplied by a decay factor. For example, if “Acquisition Decay” is -1.5 (as seen in Apex for PC, meaning maybe a 15% weekly drop if interpreted as a multiplier like 0.85 each week), we apply that to simulate declining new installs over time as the initial hype tapers.
•	We may incorporate seasonal bumps: If the Season sheet or an Events table dictates a bump at certain points, the new user count for those weeks will include the extra users. E.g., if in Season 2 Week 1 there’s a content launch, we might add a fixed number or percentage increase to that week’s new users.
•	If we consider multiple acquisition channels (like organic vs paid), we might have columns for each and sum to total new users. In early stages, user acquisition could heavily rely on marketing (paid UA), which might be capped by budget – however, since we’re not given marketing spend details here, we may keep it simpler and treat new users as an aggregate with an implicit assumption of marketing effort included.
•	Returning Users: If the Sarge model’s “Return” rate (14% in control panel) is meant for users who had churned and come back with new content, we will incorporate that logic. One way is: each time a new Season or major update starts, take a percentage of the “cumulative churned users” and add them as “returning users” in that week. Alternatively, have a modest ongoing returning user flow each week (e.g., each week a certain % of all previously lost users return, which creates a long tail of re-engagement). We’ll likely implement it by marking specific event weeks where return spikes happen (e.g., if Return = 14%, perhaps 14% of total lost users come back during a big update week). This can be part of the Events effects rather than a constant each week, since typically players return when there’s new content.
•	Active User Calculation: While new users tell us how many are coming in, we need to calculate how many active users exist in each week. This involves retention:
•	We will maintain columns for Cohort Retention or a simpler aggregate retention. A detailed way is to simulate cohorts: each week’s new users are a cohort, and we apply a retention curve to see how many of that cohort are still active in subsequent weeks. However, a full cohort model can be heavy in Excel (but feasible over a few years weekly).
•	A simpler approach (if cohorts are not absolutely required) is to use a rolling retention multiplier: e.g., “active users this week = active users last week minus some who churn plus any new and returning users”. But to stay accurate, a cohort model is better because churn rates change as cohorts age.
•	Given the importance of retention in gaming, we may do a lightweight cohort: e.g., have a fixed D7, D30 retention and assume an attrition curve beyond that (like a weekly survival rate).
•	Perhaps use the known retention: If starting retention (maybe D7 or first week retention) is 25%, that implies out of Week1 new users, 25% continue to week2. Then perhaps retention from week to week might stabilize at some percentage.
•	We can use an exponential decay or a fixed weekly retention percentage (e.g., assume each active user has X% chance to stay to next week). For initial simplicity: assume a weekly retention rate like 80% per week for the average user (just as an example), which would yield a certain decay over time. This can be adjusted for scenario (High scenario might have higher retention).
•	The Retention sheet from Apex likely had these rates, and we can incorporate such a table (maybe in Key Parameters or a hidden calc sheet). For instance, define retention % for week1->2, week2->3, etc., or monthly retention per month of life.
•	Using those, calculate for each cohort of new users how many remain active each subsequent week and sum across all cohorts to get total active users in a given week.
•	The result of this will be a column for Weekly Active Users (WAU) or since we are doing weekly, WAU might simply be the active users that week. If we want DAU, we could assume a certain % of WAU log in daily, or use WAU as a proxy (or multiply WAU by frequency factor to get total daily sessions if needed for ARPDAU).
•	We will also calculate Monthly Active Users (MAU), likely on the monthly summary sheet by taking the unique users active in a month. But since we have weekly, an approximation is to take the max of 4-week active or something. We might just report WAU or an average DAU for simplicity, depending on how monetization is modeled (often ARPDAU is easier if DAU is known).
•	Link to Monetization: The output of this sheet – particularly the number of active users each week (and possibly the number of new payers) – will feed into the monetization sheets. For example, if we know how many active users are in week 10, and what percentage of them make a purchase that week, we can compute that week’s revenue. Alternatively, if we compute an ARPDAU, we multiply by DAU. We will structure it so that either approach can be taken. Likely, we’ll have a column for Payer Conversion (number of paying users in that week) equal to active users * conversion rate (which might diminish over time or increase as the game matures and hardcore players remain). We can draw conversion rates from assumptions (e.g., 5% of actives pay in the first month, perhaps trending upward later).
In summary, the Acquisition & Growth section provides the core user funnel over time: new users -> active users -> paying users. It will be the engine that drives the rest of the forecast, and it will be sensitive to scenario inputs (higher growth in High scenario, lower in Low, etc.).
3. Monetization Modules (Revenue Streams)
Instead of one giant sheet, we will create separate sheets (or sections on one sheet) for each major revenue stream, similar to how both existing models do. This keeps calculations organized and allows tweaking each stream independently. The major revenue streams for Sarge Universe likely include:
•	Battle Pass Revenue: This sheet will calculate revenue from the Sarge Pass (and possibly the premium “Commander” pass if applicable).
•	Inputs needed: Price of the pass (e.g. £4.99 or $6.99 monthly for base pass; and £19.99 for premium if included), the conversion rate (percentage of active users who purchase the pass), and timing (battle passes might start only after launch month or every season).
•	Likely approach: For each month (or week), determine the number of battle pass subscribers. If it’s a monthly subscription model, we might assume at the start of each month X% of actives buy the pass. 4. In-Game Currency Economy (Credits Flow)
To maintain the focus on the game’s internal economy without tokens, we’ll dedicate a sheet to model Credits circulation. This is more for game design/economy health, but including it demonstrates a comprehensive simulator.

•	Sources of Credits: Typically, sources are:
•	Purchased credits (from real money – this ties to the revenue, e.g., if $1 buys 100 credits, and players spent $X on credits, then X*100 credits entered the economy).
•	Credits earned through gameplay (rewards from missions, daily quests, battle pass free track, etc.).
•	Perhaps credits given in promotions or events.
•	Sinks of Credits: Places where credits are spent or removed:
•	Buying items in the shop using credits.
•	Opening lootboxes with credits.
•	Unlocking features, speeding up progress (if the game allows spending credits for convenience).
•	Any conversion of credits back to premium currency (unlikely, since credits themselves are the premium currency presumably).
•	Modeling Approach: We will likely do a high-level model:
•	We know how many credits are bought (directly from the revenue: e.g., if $100k was spent on credits in a week, and pack prices give an average rate, we can estimate credits purchased).
•	We can estimate how many credits are earned in-game by active users (e.g., on average an active user earns 50 credits per day through gameplay – this can be an assumption).
•	Then total credits injected = purchased credits + earned credits.
•	For sinks, we assume players spend most of what they earn, plus whatever they bought, on various sinks. If we assume an equilibrium in the long run, we might set sinks such that total credits spent = total credits generated (with some lag maybe). Early on, players might stockpile some credits, so not all are sunk immediately.
•	We might model credit balance over time, i.e., cumulative credits in circulation minus spent. If this balance grows unchecked, it could indicate potential inflation (players hoarding currency). The goal in design would be to have sinks that encourage spending.
•	Key metrics we can output: Total credits circulating among players, and ensure it doesn’t explode. If it does, maybe the game would introduce more sinks or we adjust assumptions.
•	Sinks can be broken down (if needed) into categories like: % spent on lootboxes, % on speeding up build timers, % on cosmetics, etc., but that might be too granular. At least a total sink each week.
•	No Token, Just Credits: We reiterate that credits are not blockchain tokens but normal in-game currency. So we won’t model any token-related behaviors (no token market cap, no token issuance). This simplifies things: credits have a fixed value (or rather, they are only valuable in-game and purchased at a fixed exchange rate from real money).
•	This sheet might not feed into the financial summary (since it’s not about revenue) but serves as a reference for game economy balance. def extract_formula_edges(wb):
    # Build a quick map of sheet dependencies based on cross-sheet references inside formulas
    sheet_names = set(wb.sheetnames)
    dep_counts = Counter()
    formulas_per_sheet = Counter()
    sample_formulas = defaultdict(list)

    # Regex to capture potential sheet references like 'Sheet Name'!A1 or SheetName!A1
    # We will post-filter by checking if the captured sheet exists
    ref_pattern = re.compile(r"(?:'([^']+)'|([A-Za-z0-9_ ]+))!\$?[A-Za-z]{1,3}\$?\d+", re.IGNORECASE)

    for ws in wb.worksheets:
        for row in ws.iter_rows(values_only=False):
            for cell in row:
                v = cell.value
                if isinstance(v, str) and v.startswith("="):
                    formulas_per_sheet[ws.title] += 1
                    if len(sample_formulas[ws.title]) < 5:
                        sample_formulas[ws.title].append(v[:200])
                    # find references
                    for m in ref_pattern.finditer(v):
                        s1, s2 = m.groups()
                        ref_sheet = s1 if s1 is not None else s2
                        if ref_sheet and ref_sheet in sheet_names and ref_sheet != ws.title:
                            dep_counts[(ws.title, ref_sheet)] += 1
    return dep_counts, formulas_per_sheet, sample_formulas

## 4) Scenario logic pattern
- Control_Panel has side-by-side Low, Base, High. Applied column uses INDEX over a Scenario_List keyed by `Scenario_Sel`.

Example:
```
=INDEX(CHOOSE({1,2,3}, [@[Low]], [@[Base]], [@[High]]),
       MATCH(Scenario_Sel, {"Low","Base","High"}, 0))
```
For drivers stored in a matrix table, use XLOOKUP on the scenario header row.

## 5) Acquisition curve and events integration
- Base weekly new users:
```
=LET(
  w, [@WeekNumber],
  start, applied_launch_week_new_users,
  decay, applied_acq_decay_rate,
  floor, applied_acq_floor_per_week,
  MAX(floor, start*POWER(1+decay, w-1))
)
```
- Season override: XLOOKUP `w` into `tblSeasons` to retrieve season-specific `Acquisition_Start` and `Acquisition_Decay_Mod`, then recompute for weeks in that season.
- Event bumps: additive and multiplicative, for example:
```
NewUsers_Total = NewUsers_Base
               + SUMIFS(tblEvents[Acq_Bump_Abs], tblEvents[Week], w)
```

## 6) Cohort engine
- Survival vector stored in a row named `applied_weekly_survival_vec` with elements s1, s2, s3...
- Cohort active in week t for a cohort born at week c:
```
CohortActive(t,c) = CohortSize(c) * PRODUCT(1 + survival[k]) for k = 1 to (t - c)
```
Efficient Excel implementation per row:
- Place survival vector in row 1 of the sheet as a Table `tblSurvival`.
- For cohort row c and reporting week column t:
```
=IF(t < c, 0,
    [@[NewUsers]] * PRODUCT(1 + INDEX(tblSurvival[@], 1, 1) :
                                  INDEX(tblSurvival[@], 1, t-c))
)
```
To avoid PRODUCT across variable ranges becoming heavy, precompute a cumulative product row:
```
CumSurv[j] = PRODUCT(1 + survival[1..j])
CohortActive(t,c) = NewUsers_c * INDEX(CumSurv, t-c)
```
WAU_Base at week t is SUM over c of CohortActive(t,c). Then:
```
WAU_Final = WAU_Base + ReturningUsers_t
```
Where ReturningUsers_t:
```
=ROUND(applied_return_pct_on_event * Cum_Churn_To_Date, 0)
```
triggered by an event flag on `tblEvents`.

## 7) Payers and per-stream payers
- Cap conversion to WAU to avoid exceeding 100 percent:
```
Store_Payers = MIN(WAU_Final, WAU_Final * applied_store_conv_rate)
Loot_Payers  = MIN(WAU_Final, WAU_Final * applied_loot_conv_rate)
Pass_Buyers  = IF(IsPassWeek, MAU_Eligible * applied_bp_take_rate_monthly, 0)
```
- Optional overlap control: introduce `overlap_factor` inputs if needed. Default assumes independent small percentages where double counting does not breach caps. QA flags if `(Store_Payers + Loot_Payers) > WAU_Final`.

## 8) Monetization formulas
- Store revenue weekly:
```
Store_Revenue = Store_Payers * applied_store_arppu
* (1 + Event_Revenue_Lift_Total_Pct) * (1 + Event_Revenue_Lift_Store_Pct)
* (1 + Season_Monetization_Lift_Pct)
```
- Loot revenue: same structure.
- Battle pass monthly:
```
Pass_Sold_Base = MAU_Eligible * applied_bp_take_rate_monthly
Pass_Revenue   = Pass_Sold_Base * applied_bp_price
Premium_Units  = Pass_Sold_Base * applied_bp_premium_share
Premium_Revenue = Premium_Units * applied_bp_premium_price
Total_BP_Revenue = Pass_Revenue + Premium_Revenue
```
Allocate to week 1 of each month or prorate evenly over the month, controlled by a toggle on Control_Panel.

## 9) Credits economy
- Purchased credits:
```
Credits_Purchased = (Store_Revenue + Loot_Revenue) * applied_credits_per_currency_unit
```
- Earned credits:
```
Earned_Credits = WAU_Final * applied_earned_credits_per_active_per_week
```
- Sinks:
```
Credits_Spent = (Credits_Purchased + Earned_Credits) * applied_sinks_share_total
Net_Credits_In_Circ = Previous + Credits_Purchased + Earned_Credits - Credits_Spent
```
- Inflation warning if `Net_Credits_In_Circ` 4-week moving average growth rate exceeds a threshold input.

## 10) Summaries and rollups
- Weekly summary pulls totals by stream and ARPU_week:
```
ARPU_week = IF(WAU_Final=0, 0, Total_Revenue / WAU_Final)
```
- Monthly summary aggregates with SUMIFS by calendar month key:
```
SUMIFS(Weekly[Total_Revenue], Weekly[MonthKey], thisMonthKey)
```
- MAU approximation options: average of WAU across weeks in month, or max WAU in month. Controlled by a drop-down on Control_Panel.

## 11) Charts
- Monthly stacked area: Battle Pass, Store, Loot
- Lines: WAU and MAU trend; ARPU and payer conversion trend
- Optional bridge: TAM vs Year 1 run rate if `TAM_Value` input is provided

## 12) QA checks
- Cross-foot: sum of weekly by stream equals monthly by stream. Variance must be zero. Flag with conditional formatting if non-zero.
- Bounds:
  - 0 ≤ conversion rates ≤ 1
  - ARPPU ≥ 0
  - WAU_Final ≤ CumulativeInstalls + ReturningUsers cumulative
  - Payers_Store + Payers_Loot ≤ WAU_Final
- Scenario sensitivity mini-table:
