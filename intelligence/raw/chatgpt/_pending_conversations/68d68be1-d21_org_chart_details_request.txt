SLOs = **Service Level Objectives**.

Quick context (so we’re aligned):
- **SLI (Service Level Indicator):** the measurement (e.g., percentage of crash-free sessions).
- **SLO (Service Level Objective):** the **target** for that measurement (e.g., ≥99.5% crash-free sessions over 30 days).
- **SLA (Service Level Agreement):** an **external, contractual** promise (with penalties). Only set SLAs once SLOs are consistently met.

### Suggested SLOs for Couch Heroes (live game + platform)
Each line shows **SLI → SLO** (window) and notes.

- **Crash-free sessions → ≥99.5%** (rolling 30d). Error budget = 0.5% of sessions may crash.
- **Matchmaking success rate → ≥98.0%** (30d). Excludes user cancels and timeouts above a set threshold.
- **Median/95th queue time → P50 ≤ 45s, P95 ≤ 120s** (7d). Per region; excludes maintenance windows.
- **Gameplay server availability → ≥99.9%** (30d). Region-weighted; excludes planned maintenance.
- **MTTR for P1 incidents → ≤30 minutes** (quarter). Measured from page to full recovery.
- **Release health → ≥95% build success** (30d) and **rollback ≤10 minutes** (per incident).
- **Payments auth success → ≥97.5%** (30d). Provider-normalized; excludes user declines.
- **Telemetry ingestion success → ≥99.9%** (30d). Events end-to-end; sampled for integrity.
- **Content publish success (events/shop) → ≥99.0%** (30d). With automated preflight checks.
- **Anti-cheat false positive rate → ≤0.10%** (quarter). With appeal turnaround ≤24h.

### How to use them (plain English)
1. **Pick SLIs that reflect player experience** (crashes, matchmaking, latency, payments).  
2. **Set SLOs** that are ambitious but reachable with current staff (raise them later).  
3. Track a **monthly error budget**: if the SLO is 99.5%, the budget is 0.5% “allowable bad.”  
4. **Policy:** If you burn >50% of budget mid-window, freeze risky releases and focus on reliability until you’re back on track.

### Example SLO write-up (copy/paste)
- **Name:** Crash-Free Sessions  
  **SLI:** (Total sessions − Crash-ended sessions) ÷ Total sessions  
  **Objective (SLO):** ≥99.5% over rolling 30 days  
  **Scope:** Production, all regions; excludes scheduled maintenance  
  **Budget:** 0.5% (e.g., with 1,000,000 sessions/month → ≤5,000 may crash)  
  **Alerts:** Budget burn-rate >2× for 1h = page on-call; >50% budget consumed before day 15 = feature freeze  
  **Owners:** Client Eng Lead + QA Automation Lead  
  **Dashboards:** Crash rate by build, region, device; top 5 crash signatures auto-ticketed
