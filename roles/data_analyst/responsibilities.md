# Data Analyst — Responsibilities

## Job Description

The Data Analyst provides the analytical horsepower that NBI's decision-makers and clients need to act on data rather than instinct. This role supports four distinct stakeholders: the COO (delivery and operations tracking), the CFO (financial modelling and revenue analysis), the CMO (pipeline analytics and marketing performance), and NBI clients where analytical work is part of the engagement scope.

NBI has a genuine analytics heritage. Glen Pryer spent 20 years building analytics teams at Blizzard, EA/DICE, Jagex, and Build a Rocket Boy. The Lighthouse Studios engagement has Amir Didar, Ruan, and Stavros embedded as a full analytics team. The Data Analyst operates at the layer above raw engineering — taking data that exists and turning it into decision-ready intelligence.

For internal NBI work, the key analytical domains are financial (revenue tracking, payroll modelling, cost projections), pipeline (lead stage, deal progression, revenue forecasting), and delivery (client project health, milestone tracking, resource utilisation). For client-facing work, the analytical domains are gaming-specific: player segmentation, DAU/MAU forecasting, churn prediction, sentiment analysis, and monetisation modelling — the same types of work NBI delivers to studios like Lighthouse.

The Data Analyst does not build data infrastructure from scratch or run data engineering pipelines. That is an engineering function. The Data Analyst works with data that exists — whether in spreadsheets, QuickBooks exports, ClickUp project data, or Supabase databases — and produces analysis, dashboards, and models from it.

## Core Responsibilities

1. Build and maintain internal dashboards tracking NBI's key operational metrics: revenue by client, payroll costs, pipeline by stage, delivery health by project
2. Support CFO with financial models, including payroll projections, revenue forecasting, and the NSI transition cost modelling
3. Support CMO with pipeline analytics: lead conversion rates, deal velocity, revenue projection by pipeline stage, source attribution
4. Support COO with delivery tracking: milestone completion rates, project health by client, resource allocation
5. Deliver client-facing analytical work as scoped in individual client engagements (player segmentation, DAU/MAU forecasting, churn modelling, IAP/monetisation analysis, sentiment analysis)
6. Produce ad hoc analysis in response to specific decision needs from any of the above stakeholders
7. Flag data quality issues before they contaminate reports or models — never deliver analysis built on known-bad data without explicitly flagging the problem

## Key Performance Indicators

| KPI | Target | Measurement |
|---|---|---|
| Stakeholder satisfaction with deliverable quality | No report returned for fundamental rework | Rework request rate |
| Analytical accuracy | Zero factual errors in financial models or client deliverables (confirmed by review) | Post-delivery audit |
| Turnaround on ad hoc requests | 80% of ad hoc requests delivered within agreed SLA | Request log timestamps |
| Dashboard freshness | Internal dashboards updated within 24 hours of new data availability | Last-updated timestamps |
| Data quality flag rate | Every known data quality issue flagged before delivery | Escalation log |

## Interfaces

- **Receives from:** COO (delivery tracking requests, project data), CFO (financial model requests, QuickBooks exports), CMO (pipeline data, marketing performance queries), clients (raw data for gaming analytics engagements)
- **Delivers to:** COO (delivery dashboards, project health summaries), CFO (financial models, variance reports), CMO (pipeline dashboards, performance reports), clients (analytics reports, segmentation outputs, forecast models)
- **Tools:** Excel/Google Sheets (financial models, data manipulation), QuickBooks (financial data source), ClickUp (delivery tracking source), Supabase (Playsage data for product analytics), Python or SQL for larger analytical tasks, dashboard tooling as established per project

## What "Done" Looks Like

- Every analysis has a clear headline finding, not just a table of numbers
- Methodology and data sources are documented so the analysis can be reproduced or challenged
- Data quality issues or estimation caveats are visible, not buried
- The recipient can act on the output without needing a walkthrough from the analyst
- Financial models have been sense-checked for mathematical correctness before delivery
