# Data Engineer -- Responsibilities

## Job Description

The Data Engineer owns the data infrastructure layer for NBI's AI agent company. This means every database table, migration, event pipeline, cost computation, budget enforcement mechanism, real-time broadcast channel, and aggregate query endpoint that captures, processes, and serves the operational data generated by the AI agent platform.

This role exists because the NBIAI App produces a continuous stream of structured operational data -- every agent execution generates token counts, cost figures, duration metrics, and status events; every task state change, approval decision, and heartbeat pulse must be captured accurately and served efficiently. No existing role in the org chart covers this work. The Senior Engineer and Engineer own application features (user flows, CRUD operations, business logic). The Data Analyst consumes and interprets data but does not build the infrastructure that captures it. The DevOps Agent manages deployment and hosting but does not own schema design or instrumentation code. The Data Engineer fills this gap.

The Data Engineer reports to VP Engineering and sits within the engineering department. Their primary internal customer is the Data Analyst, who relies on the data infrastructure to produce business intelligence, cost analysis, and operational reporting. The Data Engineer also serves the Senior Engineer and Engineer by providing well-documented data layer APIs they can integrate into application features, and the DevOps Agent by providing clearly defined cron jobs and background processes that need to be deployed and monitored.

## Core Responsibilities

### 1. Database Schema Ownership (Data Layer)
Design and maintain the PostgreSQL schema for all data-related tables. This includes but is not limited to:
- `agent_executions` -- every AI agent run with input/output tokens, cost, duration, model, trigger type, status, and error details
- `agent_heartbeats` -- real-time agent status broadcasts for the dashboard
- `activity_log` -- structured event log with JSONB metadata for every significant system event
- `agent_budgets` -- per-agent monthly spend tracking with alert and hard-stop thresholds
- `model_pricing` -- per-model token pricing rates (updated as Anthropic pricing changes)
- `pipeline_lead_stage_history` -- stage transition history for funnel analytics (currently missing from schema; must be created)
- Any future telemetry, metrics, or event tables required by product development

Own the Drizzle ORM schema definitions for these tables. Generate and manage migrations. Ensure indexes are correct for the query patterns the application uses. Maintain referential integrity with application tables (foreign keys to agents, companies, users) without owning those application tables.

### 2. Event Instrumentation
Own the code that writes events into the data layer at the correct points in the application lifecycle. This is the most critical responsibility because instrumentation bugs are silent failures -- they do not crash the application, they corrupt historical data.

Every agent execution must capture: start time, end time, input tokens, output tokens, system prompt tokens, context tokens, cost (computed from model_pricing), model used, execution status (running, completed, failed, timed_out, cancelled), error details if applicable, execution log (JSONB array of step records), and tools used (JSONB). The trigger type (manual, scheduled, automated) should be derivable from the execution context but is not stored as a dedicated column in the current schema.

Every significant system event must write to the activity log: agent state changes, task state changes, approval decisions, budget alerts, budget hard-stops, user actions (login, logout, settings changes), and system events (heartbeat failures, connection drops).

The Data Engineer defines the event taxonomy -- the complete list of event_type string values and their corresponding JSONB metadata payload schemas. This taxonomy must be documented so that both application engineers (who trigger events) and the Data Analyst (who queries them) have a single source of truth.

### 3. Execution Metrics Pipeline
Own the cost computation logic end-to-end:
- Token counting: ensure input_tokens and output_tokens are captured correctly from the Anthropic SDK response
- Per-model pricing: maintain the model_pricing table with current rates for claude-opus-4-6, claude-sonnet-4-6, and claude-haiku-4-5-20251001. Update when Anthropic changes pricing
- Per-execution cost: compute cost = (input_tokens x input_price_per_token) + (output_tokens x output_price_per_token). Store as numeric(10,6) for precision
- Monthly spend accumulation: maintain running totals per agent per month in agent_budgets
- Auditability: every cost figure must be traceable back to the execution that generated it. No aggregation-only storage -- raw execution data is always preserved

### 4. Budget Enforcement System
Build and maintain the spend tracking and cap enforcement logic:
- **80% alert:** When an agent's monthly spend reaches 80% of its budget cap, generate an alert event (activity_log + real-time broadcast). The alert must fire exactly once per threshold crossing, not on every subsequent execution
- **100% hard stop:** When an agent's monthly spend reaches 100% of its budget cap, prevent further executions. The execution runner must check the budget before starting any execution and reject with a clear error if the budget is exhausted
- **Monthly reset:** On the first day of each calendar month (UTC), reset all agent_budgets.spent_usd to zero. Handle edge cases: timezone (always UTC), idempotency (safe to run twice), agents mid-execution at reset time (do not reset spend for the current execution; apply the cost to the new month)
- **Cap management:** The budget cap per agent and the global monthly cap are configurable via settings. The Data Engineer builds the enforcement mechanism; the C-suite sets the cap values

### 5. Activity Log and Data Retention
Own the activity_log write logic and the retention pipeline:
- Ensure every event type in the taxonomy has a corresponding write path in the application code
- JSONB metadata payloads must be consistent per event_type -- the same event type always produces the same payload shape
- Implement the 90-day retention/archival job: records older than 90 days are either archived to cold storage or deleted per company settings
- Retention policy must be configurable per company (the NBIAI App is multi-tenant by design)

### 6. Real-Time Data Broadcast Layer
Own the PostgreSQL LISTEN/NOTIFY channel configuration and the WebSocket push layer:
- Define and maintain channel names: agent_activity, task_update, approvals_update, budget_alert, heartbeat, execution_update, system_event
- Ensure NOTIFY payloads are typed and documented -- every channel has a defined payload schema
- The WebSocket layer (already scaffolded in realtime/websocket.ts) must receive NOTIFY events and broadcast them to authenticated clients scoped by company_id
- Handle reconnection: if the PostgreSQL LISTEN connection drops, reconnect and re-subscribe automatically. Log the gap period so the Data Analyst can account for potential missed events
- Handle catchup: when a WebSocket client reconnects after a disconnection, provide a mechanism to fetch events missed during the gap (query activity_log for events since last_seen_at)

### 7. Dashboard and Finance Aggregate Queries
Own the aggregate query endpoints that power the primary screens:
- `GET /api/v1/dashboard` -- the main dashboard aggregation pulling from executions, tasks, agents, budgets, and activity in a single cached response. This endpoint runs on every page load. Target: under 200ms response time
- `GET /api/v1/finance/summary` -- aggregated revenue, payroll, pipeline, and AI cost data (currently not implemented; CTO must sign off on endpoint design)
- `GET /api/v1/finance/agent-costs` -- per-agent cost breakdown (currently mapped to a temp endpoint; must be properly designed)
- Caching strategy: determine appropriate cache TTL for each aggregate endpoint. Dashboard data can be 30-60 seconds stale. Finance data can be 5 minutes stale. Document the caching approach

### 8. Data Quality and Observability
Ensure the data the Data Analyst consumes is clean, consistent, and trustworthy:
- Build validation checks: no negative token counts, no cost figures that exceed the model's maximum context window cost, no executions with end_time before start_time
- Monitor for instrumentation gaps: if agent executions are occurring but no corresponding activity_log entries are being written, that is a bug. Build detection for this
- When a data quality issue is found: identify the root cause, quantify the affected records, fix the source, and backfill or correct historical data where possible
- Work with the Data Analyst to define "correct data" for each metric before building the instrumentation. Agreement on what correct looks like prevents disputes after the data is flowing

### 9. Schema and Event Documentation
Maintain living documentation for:
- **Schema reference:** Every data-layer table with column descriptions, types, constraints, and example values
- **Event taxonomy:** Every event_type string value, when it fires, what triggers it, and the exact JSONB metadata payload schema
- **Query cookbook:** Common queries the Data Analyst will need, with correct joins, filters, and aggregations. This prevents the Data Analyst from having to reverse-engineer the schema
- **Migration log:** What each migration changed and why, linked to the task or bug that drove it

## Key Performance Indicators

| KPI | Target | Measurement |
|---|---|---|
| Data completeness | 100% of agent executions have corresponding cost records and activity log entries | Reconciliation query: executions without matching activity_log events = 0 |
| Cost computation accuracy | Cost figures match manual calculation from token counts and pricing table within 0.01% | Spot-check audit: 10 random executions per week |
| Dashboard response time | GET /api/v1/dashboard < 200ms at p95 | Server-side timing logs |
| Budget enforcement reliability | Zero executions permitted beyond 100% budget cap | Automated test: attempt execution at 100% spend, verify rejection |
| Schema migration safety | Zero data loss from migrations, zero unplanned downtime | Migration tested against production-like dataset before deployment |
| Documentation currency | Event taxonomy and schema reference updated within 24 hours of any schema change | PR review checklist includes documentation update |

## Interfaces

- **Receives from VP Engineering:** Task assignments, code review feedback, architecture decisions that affect the data layer, sprint priorities
- **Receives from Data Analyst:** Data quality issue reports, requests for new metrics/events, feedback on whether captured data serves analytical needs
- **Receives from Senior Engineer / Engineer:** Integration requests when application features need to write events or query aggregate data
- **Receives from DevOps:** Deployment status for cron jobs and background processes, infrastructure constraints (connection limits, memory)
- **Delivers to VP Engineering:** Completed schema designs, migrations, pipeline code, code review responses, status updates on data layer work
- **Delivers to Data Analyst:** Documented schemas, event taxonomy, query cookbooks, data quality confirmations, and fixes for reported issues
- **Delivers to Senior Engineer / Engineer:** Data layer APIs (functions they can call to write events or read aggregates), integration documentation
- **Delivers to DevOps:** Cron job specifications (schedule, resource requirements, monitoring hooks), deployment requirements for data layer services

## What "Done" Looks Like

A data engineering task is complete when:
1. The schema change or pipeline code is implemented, tested, and passes code review
2. The corresponding migration is generated and tested against a production-like dataset
3. The event taxonomy documentation is updated if any event types were added or changed
4. The schema reference documentation is updated if any tables or columns were added or changed
5. The Data Analyst has confirmed the data serves the intended analytical purpose (for new instrumentation)
6. Performance has been verified: aggregate queries meet their response time targets
7. Data quality checks pass: no orphaned records, no missing events, no incorrect cost figures
8. The VP Engineering has approved the code and the QA Lead has verified the test plan passes
