# UI/UX Designer — Responsibilities

## Job Description

The UI/UX Designer is the production execution layer of NBI's design function. Where the UI/UX Lead sets strategy, defines components, and owns the design system, the UI/UX Designer creates the actual assets: icons, illustrations, component variants, prototype screens, and visual assets for the Playsage platform and the NBI/PlaySage website. This role exists to increase the throughput of the design function by handling high-volume, high-craft execution tasks under the UI/UX Lead's direction.

The UI/UX Designer does not set design direction, alter the design system, or make brand decisions independently. Every deliverable is produced against a brief provided by the UI/UX Lead. The quality bar is execution fidelity to the brief, brand consistency, and Engineer-ready output.

NBI's design language is gaming-industry-native: dark theme, electric blue accents, Orbitron headings, high information density, cinematic aesthetic. The UI/UX Designer must internalise this system and apply it precisely, not through personal interpretation.

## Core Responsibilities

1. Create production-ready UI assets from UI/UX Lead briefs: icons, illustrations, component variants, background textures, hero images
2. Build prototype screens and interactive mockups in Figma as directed
3. Execute component-level design work: all states (default, hover, active, disabled, error, loading, empty) for every component assigned
4. Export assets in formats required by Engineering: SVG, PNG at correct resolutions, Figma component links, CSS tokens where applicable
5. Apply design system rules consistently across all work without deviation unless directed otherwise by the UI/UX Lead
6. Maintain file hygiene in Figma: named layers, organised frames, no orphaned elements
7. Support the UI/UX Lead in building the Playsage design system by executing component documentation screens as directed
8. Flag asset creation blockers (missing icon references, undefined states, conflicting specs) to the UI/UX Lead before the work stalls

## Key Performance Indicators

| KPI | Target | Measurement |
|---|---|---|
| Brief-to-delivery cycle time | Assets delivered within agreed timeline on 90%+ of tasks | Task log timestamps |
| Revision rate | Fewer than 2 revision cycles per asset before UI/UX Lead approval | UI/UX Lead review records |
| Design system compliance | Zero brand violations in delivered assets | UI/UX Lead review |
| Engineering-ready quality | No more than 1 asset export issue raised by Engineering per sprint | Engineering feedback |

## Interfaces

- **Receives from:** UI/UX Lead (all briefs, feedback, approvals, design system documentation)
- **Delivers to:** UI/UX Lead (for review and approval before anything goes to Engineering)
- **Tools:** Figma (primary design and asset tool), Framer (website execution support where needed), export tools for SVG/PNG

## What "Done" Looks Like

- Every state defined in the brief has been designed (not just the happy path)
- Assets are exported in the correct format and named according to the design system conventions
- Figma files are clean: named layers, organised frames, no stray elements
- The UI/UX Lead can approve the output without asking follow-up questions about intent
- Engineering can implement directly from the delivered assets without needing to ask the designer what they meant
