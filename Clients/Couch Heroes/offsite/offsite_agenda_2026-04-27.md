## Day 2 - Gates, Live Services, GTM

**Outcome target:** Gate-passing criteria defined for all 8 gates. Live services and GTM lanes added to the roadmap. Community plan framed. Studio-down goals cascade defined.

### Session 1 - Gate-passing criteria
For each of the 8 gates (Concept / Pre-prod / Early prod / Mid prod / Late prod / Alpha / Beta / Launch), define:
- **What must be true to enter this gate** (entry criteria)
- **What must be true to exit this gate** (exit criteria)
- **Who decides** (gate keeper - usually a discipline lead pairing)
- **What evidence is required** (deliverable list)
- **What can slip and what cannot**

This is the single most leveraged hour of the offsite. Get it right and every future estimate has a yardstick.

### Session 2 - Re-examine the gate wall
With the criteria locked, walk back to the gate wall from Day 1. Move features that don't actually meet entry criteria for their column. Catch the ones that were placed by hope rather than evidence.

### Session 3 - Live Services - what's missing
The Live Service Features sheet has 79 items. The Feature Versions sheet has the V0-V7 maturity model up to V7=Scale. Build out the live service plan:
- What's the live service rhythm post-launch? (Seasons? Events? Patches?)
- What live ops tooling do we need built before launch (lest we ship and have nothing to operate with)?
- Which Game Features need live service hooks at architecture stage (not bolted on later)?
- Battle pass yes/no, season cadence, content calendar shape

### Session 4 - Go-to-Market lane
Per Glen's framing:
- **Player target segments** - confirm Explorer-primary, Socialiser-secondary, Achiever-tertiary, Killer-supported (per vision doc) or revise
- **Positioning vs competitor map** - Light No Fire, Pax Dei, Palia, Wynncraft, FFXIV

- **Demo / beta / early access** - what and when (open question for years pre-launch)
- **Publisher vs self-publish** - direction confirmed

### Session 5 - Community: how, when, why
- **How** - which channels, what tone, what cadence, what content

### Session 6 - Studio-down goals cascade

## Day 3 - Pipelines, RACI, Staff, Wrap

**Outcome target:** Every pipeline mapped, RACI per pipeline defined, staff plan validated against roadmap, decisions captured for Confluence rollout.

### Session 1 - Pipelines, mechanical
Map every pipeline that exists in the studio:
- **Art pipeline** - concept to in-engine asset
- **Audio pipeline** - brief to ship-ready audio
- **Code pipeline** - feature spec to merged code
- **Design pipeline** - idea to GDD to feature spec
- **QA pipeline** - feature ready to playtested to released
- **Content pipeline** - idea to in-game content (quest, level, item)
- **Live ops pipeline** - event idea to live to retro
- **Build pipeline** - commit to playable build
- **Marketing pipeline** - moment to asset to publish

For each: steps, tools at each step, handoffs, what triggers the next step.

### Session 2 - Pipelines, communication and ownership
For each pipeline, define RACI (Responsible / Accountable / Consulted / Informed):
- Who **owns** each stage (Accountable)
- Who **does the work** at each stage (Responsible)
- Who **must be consulted** (mandatory, not optional)
- Who **must be informed** (mandatory, not optional)
- Where optional consultation/info is fine, where it's mandatory

Output per pipeline: a one-page Confluence article with a swimlane diagram. Templates locked here so the team can fill them after offsite.

### Session 3 - Maintenance check
For each pipeline RACI: how often does this need updating? Who updates it? What triggers a refresh?
Overhead must be low - if maintaining the RACI takes a day a quarter per pipeline, it dies. Aim for "review at gate transitions".

### Session 4 - Staff assessment
Smaller group (C-level + directors only, leads excused for this session).
- Walk the previous quad review (weak links, loose cannons, champions, steady eddies)
- For each named individual:
  - Where does the new roadmap put pressure on them?
  - Where do we have strength we haven't been using?
  - Where do we have a gap that no current person fills?
- Identify the 3-5 highest-pain-point people and a path forward for each (coaching, role change, performance plan, replacement)
- Tie hiring plan changes back to HR project in workbook

### Session 5 - Decisions log walkthrough
Read every decision made across Days 1-3. Confirm or revise. Capture owners and follow-up dates.

### Session 6 - Confluence rollout commitment
- Each pipeline RACI -> who writes the Confluence article, by what milestone
- Each gate criterion -> Confluence article owner
- Each studio-down goal cascade item -> publication plan
- The full roadmap -> who publishes to the studio, when

## Three Days Vs The Work Load - Honest Read

This agenda is heavy. Realistically:
- **Day 1 Sessions 3-4 (Feature sweep)** is the biggest risk to overrun. 1,194 Game items can't all be reviewed individually in two sessions. The mitigation: focus on Game Features (561) and Game Systems (162) only on Day 1; Content / Platform / Live Service / Feature Versions get reviewed as part of Days 2 specific sessions where they naturally fit.
- **Day 1 Session 3** uses the existing classification + ownership data in the workbook as the starting point. The room is **validating and editing**, not creating from scratch.
- **Day 2 Session 4-6 (GTM, Community, Goals cascade)** can stretch. If they do, Day 3 Session 4 (Staff) shrinks - acceptable trade because Staff Assessment is C-level only and can move to a follow-up leadership session if needed.
- **Day 3 Session 1-3 (Pipelines)** is the second biggest risk. Mitigation: start with the **two highest-pain pipelines** (likely Art and Code) as the worked examples, the rest follow the template.

If we're falling behind by end of Day 2: Day 3 collapses Pipelines into "frame the answer + assign owners to write up after" rather than "fully define in room".
