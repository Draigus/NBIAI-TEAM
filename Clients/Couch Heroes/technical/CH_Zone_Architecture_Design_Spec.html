Zone Population Architecture

Detailed technical design specification for Couch Heroes multiplayer zone infrastructure
Client: Couch Heroes Prepared by: NBI Gaming, Glen Pryer, CPO (Advisory) Version: 3.0 Date: 4 June 2026
Audience: Vardis (CEO), Aris (COO), Mustafa (Head of Tech), Robin (Game Director)

0. Decision Framework

We need to lock in a zone population target so the tech team can start prototyping the networking layer. How many concurrent players per zone are we building for?

It is not a single number. It is a spectrum, and each tier on it carries a different engineering investment, a different team composition, and a different competitive position. The architecture below is designed so we start at the recommended tier and use prototype data to find out where the real ceiling is before committing production resources.

Population Tiers

Tier 1: 100 to 150 Players Per Zone

What this involves: The Unreal Engine 5 Replication Graph plugin must be enabled and configured. This is not a toggle. Replication Graph replaces the default O(N*M) relevancy check system with a custom node tree. Developers must subclass UReplicationGraph in C++, create and configure spatial nodes (such as UReplicationGraphNode_GridSpatialization2D), assign actor classes to appropriate nodes, and tune NetCullDistanceSquared values per actor type. They must also configure network dormancy settings per actor class and tune NetUpdateFrequency to control per-actor replication rates. This is meaningful C++ engineering work, not configuration file changes.

Team requirement: At least 1 to 2 engineers with prior UE5 Replication Graph experience. A team without this experience will spend significant pre-production time learning the system before productive work begins.

Competitive position: This tier puts Couch Heroes at parity with Fortnite's 100-player battle royale cap, which was built by Epic's own engine team with source-level access to the engine. It is below FFXIV's commonly reported zone cap of approximately 300 players. For a Social MMO where the core promise is a populated world, 100 to 150 may not feel sufficiently alive, depending on zone size and content density.

Risk level: Moderate. This tier is within proven territory for UE5 multiplayer titles, but it is not "out of the box" and requires non-trivial custom networking code.

Tier 2: 200 to 300 Players Per Zone Recommended

What this involves: Everything from Tier 1, plus a set of custom systems that do not ship with UE5 and must be built by the engineering team:

  • Custom interest management tiers: A custom Replication Graph node that varies replication frequency based on distance from the viewer. UE5 does not ship a node that does this automatically. The standard GridSpatialization2D handles spatial culling (deciding which actors are relevant) but not frequency tiers (deciding how often relevant actors replicate). The custom node must implement at least three distance tiers with separate promote and demote thresholds (hysteresis) to prevent oscillation at tier boundaries.
  • Hub sharding: When population in Downtime (the main city) exceeds a threshold, the system must spawn a parallel instance and route players to it based on their social graph (party, guild, friends list, faction). This involves a backend routing service, a Redis-based state transfer pipeline for player data during transitions, and UE5 ClientTravel() integration to move players between dedicated server processes.
  • Staggered replication: At 300 connections, the server cannot replicate to every connection on every tick. Following the pattern used in Fortnite, the server replicates to a subset of connections each tick (50 to 100) and rotates through the full set. UE5's priority accumulation system ensures that high-priority actors (combat participants) win bandwidth in every pass, while low-priority actors (distant explorers) accumulate priority over time and eventually receive updates. This is configurable through GetNetPriority() overrides and NetUpdateFrequency settings, but the staggering behaviour itself requires understanding and tuning the replication pipeline.
  • State transfer pipeline: When a player moves between servers (entering Downtime, leaving a dungeon, entering a User Space), their state must be serialised to Redis, transmitted to the receiving server, and deserialised without data loss. This pipeline requires a verification handshake where the receiving server confirms successful state loading before the source server releases the connection. If the transfer fails, the player must be returned to the source server, not dropped.

Team requirement: 2 to 3 senior engineers with UE5 dedicated server experience, dedicated to networking infrastructure. This is a multi-quarter infrastructure build, not a sprint or two of work. These engineers will be fully occupied with the networking layer during pre-production and early production.

Competitive position: This tier approaches FFXIV's commonly reported zone cap of approximately 300 (which Square Enix does not officially publish). It would be unusual among shipped Social MMOs, where most cap at 25 to 50 with gameplay systems. Social virtual worlds without combat reach higher (VRChat at 80, Second Life at 100 basic avatars per region), but they lack game systems. Achieving 200 to 300 in UE5 with real-time open-world combat running on the zone server would be a notable technical achievement.

Risk level: Moderate to High. The individual patterns are all proven in shipped titles, but no shipped UE5 game has publicly demonstrated 300 players per zone with combat. The 200-player prototype is the gate that validates whether this tier is achievable for Couch Heroes' specific actor complexity.

Recommendation

Build for 200 to 300 per zone. Prototype at 200 first. The prototype tells us where the real ceiling is before we commit production time to it.

One thing everyone needs to be clear on

Zone population and combat population are different numbers. Any single combat encounter (world boss, corruption event, PvP fight) caps at about 100 participants. Players beyond that observe but cannot join the fight. The zone holds 200 to 300 players coexisting across exploration, social, and combat. The combat happens in clusters of up to 100 within that zone. This is the standard MMO pattern and it is how we should talk about it externally.

1. Executive Summary

Our design pillars (discovery, corruption, no-stat cosmetics, soft-class, social-first) all depend on the same thing: the world needs to feel alive and full of other players. The GDD originally specced 30 to 40 per instance. That is not enough for a Social MMO. This document lays out what it takes to get to 200 to 300 per zone, with Downtime sharding dynamically as it fills up, and combat encounters running smoothly at up to 100 participants.

Why This Matters

The social experience is the product. A world with 30 players per zone feels empty. Palia is the cautionary example: its low population cap was frequently cited by players as harming the social feel, and the studio's commercial outcomes speak for themselves. Our five factions only feel real when faction districts have people in them. Corruption only generates shared stories when multiple players see it happen. The cosmetic system only drives aspiration when players see each other's gear. All of that requires density.

Social MMO competitors with gameplay systems cap at 25 to 50 players per zone (Palia, Tower of Fantasy, Destiny 2). Social virtual worlds without combat reach higher (VRChat at 80, Second Life at 100) but lack game systems. FFXIV is commonly observed around approximately 300 (not officially published by Square Enix) per zone, with combat heavily instanced on separate servers and a long-mature custom engine. A UE5 Social MMO that delivers 200 to 300 players coexisting in a zone with real-time open-world combat would be a genuine differentiator: approaching FFXIV's zone density while supporting open-world PvE and PvP on the zone server.

Lessons for Couch Heroes

The "Empty World" Failure Mode

Palia, Tower of Fantasy, and Destiny 2's Tower all demonstrate the same problem: low player density in a social space makes the world feel dead. Palia's 25-player cap was the most extreme case, directly contributing to community dissatisfaction and the studio's commercial failure. Tower of Fantasy's invisible channel splitting created the opposite problem: the world technically had players, but you never saw the same ones twice. For Couch Heroes, where social connection across five factions is the core experience, zone populations below 100 carry significant product risk.

The "Performance Promise" Failure Mode

Throne and Liberty, ESO Cyrodiil, and (to a lesser extent) GW2 WvW all demonstrate that marketing large-scale combat and delivering smooth large-scale combat are different things. Throne and Liberty's server consolidation 5 months post-launch is the starkest warning. ESO has spent over a decade trying to fix Cyrodiil's large-scale PvP performance and recently began testing simplified abilities as a workaround. The lesson: combat must be designed from the start to be networking-friendly. Retrofitting combat to work at scale is far harder than designing it for scale from day one.

FFXIV's Proven Model

FFXIV demonstrates that visible numbered hub instances are accepted by players at scale. The system ("Limsa Lominsa 1, 2, 3") is not invisible, but it is functional and the player base has adapted to it. FFXIV routes friends, party members, and Free Company members to the same instance, though the exact routing logic is not publicly documented. This is the model Couch Heroes should follow for Downtime hub sharding, with the option to make transitions less visible if prototype testing shows that seamless transitions are reliable.

GW2's Social Routing

GW2's megaserver scoring algorithm (party, guild, language, home world) is the best shipped model for social-graph-aware instance routing. The centralised Trading Post is the correct model for cross-shard commerce. Direct player-to-player trades require co-location, but the marketplace is global. Both patterns should be adopted for Couch Heroes.

3. Architecture Overview

The architecture uses five distinct server profiles, each tuned for the specific activity type it serves. The rationale for separating these into distinct server types (rather than running everything on one server) is that different activities have fundamentally different networking requirements. Social interactions are cheap (text chat, emotes, wardrobe changes require minimal bandwidth). Combat is expensive (position updates, ability activations, damage resolution, and visual effects at high frequency). Housing is persistent and low-density. Dungeons and PvP require guaranteed tick rates with small, predictable player counts. Trying to serve all of these on a single server profile forces a compromise that makes everything worse.

Game World Architecture: Five Server Profiles
1. Overworld Zone Server : 200 to 300 players, 10 to 20 Hz tick

A single UE5 dedicated server process per zone. Handles exploration, open-world combat clusters, corruption events, gathering, questing, and social interactions outside of Downtime. Uses aggressive interest management to keep per-connection workload manageable: each player only receives updates for the nearest 30 to 80 players, with distant players dormant or at very low update rates.

Player Distribution Across a 300-Player Zone Ecosystem

A "300-player zone" refers to the total ecosystem of server processes serving one game zone, not a single server process. The overworld zone server, hub shards, and instance servers each handle a portion. A plausible busy-state distribution looks like this:

  • 40 to 80 in Downtime areas (on hub shard servers, separate server processes from the overworld)
  • 30 to 80 in active combat clusters (corruption events, world bosses, open-world PvP). These are the most networking-expensive players.
  • 120 to 200 exploring, questing, gathering, fishing, travelling between points of interest. These players are spread across the zone and mostly dormant from each other's perspective.
  • 20 to 50 transitioning to or from instanced content (entering dungeons, leaving arenas, visiting User Spaces)

These are illustrative proportions, not hard allocations. The actual mix will vary by time of day, content cadence, and event scheduling.

4. Corruption Mechanic: Networking Design

Corruption is one of Couch Heroes' five design pillars. It is a persistent environmental force covering terrain, structures, and creatures, representing the physical expression of the antagonist Drystan's virus. Players fight it back through cleansing actions, quest objectives, and collaborative world events. The corruption system changes the visual appearance of zones, affects which areas are accessible, modifies enemy spawns, and drives core quest content (such as the "Restore the Lighthouse" quest).

For the networking architecture, corruption presents a specific challenge: it is world state that changes over time and must be visually consistent across all players in a zone. Unlike player combat (which only matters to nearby participants), corruption visuals affect the entire zone's appearance and must be replicated to every player in the zone.

Corruption State Model: Tiered Hybrid

Layer 1: Per-Zone Corruption Scalar

Each zone has a single authoritative corruption value between 0 and 100, stored as a single byte. This scalar drives:

  • Enemy spawn tables (higher corruption = more corrupted enemy variants)
  • Accessibility gates (certain areas become impassable above corruption thresholds)
  • Broad VFX palette shifts (the zone's overall visual tone shifts as corruption rises or falls)
  • NPC behaviour changes (merchants may flee, quest givers may offer different quests)

This scalar is replicated to all clients in periodic world-state snapshots. At one byte, the bandwidth cost is negligible. This is the authoritative gameplay value that all game systems read.

Layer 2: Low-Resolution Spatial Grid

A 64 by 64 cell grid overlaid on the zone, with each cell holding a one-byte corruption intensity value. Total grid size: 4 KB. This grid drives:

  • Visual corruption spread patterns (which specific areas of the zone are visually corrupted)
  • Localised VFX (corruption tendrils, particle effects, terrain discolouration in specific cells)
  • Corruption-modified traversal (specific paths blocked or altered by corruption in those cells)

The grid is replicated via delta compression: only cells that have changed since the last update are sent. In practice, corruption spreads slowly (it is not frame-critical), so a typical update contains 50 to 200 bytes of changed cells, sent every 5 to 10 seconds. This is usually smaller than one full moving character update under this replication model.

UE5 Implementation: A single "Corruption Manager" actor per zone, using DORM_DormantAll by default. When corruption changes (player cleansing action, time-based spread event), the manager wakes via FlushNetDormancy(), replicates the delta to all connections, and returns to dormancy. Between corruption events, this actor has negligible normal replication work, subject to profiling with Replication Graph overhead, dormancy wake-up cost, late joins, and delta serialisation.

Additional requirements: Corruption grids need versioned snapshots (so late-joining players receive the current state, not a partial delta), idempotent cell updates (so duplicate cleansing events do not corrupt the grid), and full-state recovery for players who join mid-event.

Cross-Shard Consistency

Corruption state is owned by a central world-state service backed by PostgreSQL. This service is the single source of truth for all corruption data across the game world. When a player cleanses corruption in the overworld zone, the change propagates to the central service, which fans out to all Downtime hub shards and any other zone instances via pub/sub. Latency of 1 to 5 seconds between the cleansing action and the state being visible on other shards is acceptable for environmental state. Players will not notice this propagation delay because corruption visual changes are gradual, not instant.

5. User Space (Housing) Architecture

The GDD describes User Spaces as personal pocket dimensions accessed via the House of Houses in Downtime. Features include furniture placement, trophies, curated displays, social rooms, gardens, Batsuit storage (stored loadouts for instant class switching), and visitor portals. User Spaces are identity-first spaces, not sandbox UGC.

Server Model

User Spaces are hosted on shared housing server processes. The initial sizing hypothesis is 20 to 50 concurrent occupied User Spaces per server process using UE5 World Partition streaming. This must be validated with furniture count, visitor count, memory, serialisation, and persistence tests. Only sub-levels where players are actively present are loaded into memory. When all housing servers reach capacity, new server processes are spawned on demand by Agones. When a User Space has no visitors and the owner leaves, the sub-level is unloaded.

This matches ESO's approach, where all player homes are instanced (no land scarcity, no competition for housing plots). FFXIV's ward-based system with limited plots creates artificial scarcity that generates player frustration. The instanced model avoids this entirely.

Visitor Cap and Constraints

12 to 24 visitors per User Space, consistent with ESO (12 for standard homes, 24 for manors) and FFXIV's apartment model. The real constraint on housing instances is not player count but furniture state. Static furniture should be persisted placement data loaded on entry, not live-replicated actors. The actual server load comes from interaction state (chairs being sat in, crafting stations being used), serialisation of placement data on entry, and load time for complex User Spaces. The furniture cap should be determined by prototype testing with realistic furniture counts, interaction patterns, and visitor numbers. 150 to 300 placed items per User Space is a reasonable starting hypothesis based on shipped precedent.

Zone Server Interaction

Entering a User Space is a server transition. The player's client disconnects from the current zone or hub shard server and connects to the housing server via ClientTravel(). Player state (inventory, appearance, location metadata) is serialised to Redis before transition and loaded by the housing server. The transition is masked by a portal animation (walking through the House of Houses portal). Expected transition time: 3 to 5 seconds, consistent with other server transitions in the architecture.

Hardware Profile

Housing servers are lightweight. No combat, no AI, no large-scale interest management, no staggered replication. A c7i.xlarge instance (4 vCPU, 8 GB RAM) can host 20 to 50 occupied User Spaces comfortably.

6. Combat Networking Design

The 100-Participant Event Cap

Open-world events (corruption cleansing, world bosses, faction territory defence) cap at 100 active participants. This is a networking constraint that must be designed as a game mechanic, not hidden as a technical limitation.

How It Works

  • When a world event is triggered, a participation zone is defined with a specific radius.
  • The first 100 players who enter the participation zone and engage (deal damage, heal, use a cleansing ability) become active participants. They receive full combat-priority networking (elevated NetPriority of 3.0, higher effective update rate).
  • When the event reaches 100 active participants, it transitions to a "full" state. Additional players who arrive within the event radius become observers.
  • Observers can see the event happening (at reduced visual fidelity, lower update rate) but cannot deal damage or receive loot. The observation radius extends beyond the participation radius, so the spectacle is visible to passersby.
  • When an active participant leaves (dies and respawns elsewhere, runs away, disconnects), a slot opens. The longest-waiting observer is promoted to active participant.

Game design framing: Robin's team should frame this as "the corruption is overwhelming; reinforcements standing by" or "the battle is at capacity; hold the perimeter." The 100-player cap is a feature, not a bug. It exceeds most shipped MMO combat encounters (FFXIV caps at 24 for alliance raids, 72 for PvP Frontlines).

Soft-Lock Hybrid Combat Model

The soft-lock hybrid combat model was selected because it provides the best balance between networking efficiency and player skill expression. In this model, target selection is assisted (the game helps you lock onto a target within range and line-of-sight), but skill expression comes from what you do after locking: positioning, timing, combo sequencing, dodge management, and lock maintenance.

Why Soft-Lock is Right for This Architecture

  • Server-authoritative validation is simple and binary. The server validates: "Was the target within lock-on range at the server-authoritative timestamp?" This is a binary yes/no check that does not require precise hit detection, projectile simulation, or favour-the-shooter lag compensation. It works cleanly at high latency and with staggered replication.
  • Bandwidth per combat interaction is lower than full action combat. There are no projectile trajectories to replicate, no frame-precise hit boxes to synchronise. The lock state, ability activation, and damage result are compact messages.
  • Client-side prediction works cleanly. The player's ability activation is predicted locally (animation plays immediately). The server validates the lock and resolves damage. Corrections are rare because the lock constraint is binary, not probabilistic.

Skill Expression Techniques

Soft-lock does not mean "point and click." The following techniques, all shipped in production titles, create genuine skill depth within the soft-lock model:

Positioning and Movement

  • Positional bonuses: Flanking and rear attacks deal bonus damage. Standard in FFXIV (Monk and Dragoon positional abilities). Forces players to manoeuvre during combat, not stand still.
  • Dodge i-frames: Invincibility frames during dodge with precise timing windows. GW2's primary skill expression mechanic. Rewards reaction time and pattern recognition.
  • Momentum-preserved dodges: Movement speed and direction at the moment of dodge input affect the dodge trajectory. Adds depth to positioning decisions.
  • Split-body animation: Upper body locked to attack target, lower body free to move and dodge independently. Demonstrated in Ashes of Creation's combat prototype. Allows simultaneous attacking and repositioning.

Lock Management and Combos

  • Lock degradation: The lock weakens or breaks when the target dashes, enters cover, or exceeds maximum range. Creates a skill gap between players who maintain lock through target movement and those who lose it.
  • Combo sequencing: Ability chains that reward correct order with bonus effects. Adds decision-making depth to every combat rotation.
  • Combo fields and finishers: Abilities that interact when overlapping spatially. GW2's combo system creates emergent depth from combining different players' abilities.
  • Animation cancelling windows: Allowing players to cancel recovery frames of one ability into the startup of another creates an optimisation skill ceiling.

Networking-Friendly Combat Design Rules

These rules must be established before any ability is implemented. They are game design constraints driven by networking requirements.

  • AoE abilities cap at 12 targets (nearest by distance). This reduces combat fan-out and helps avoid O(N squared)-style worst cases when many players cluster. Without this cap, a single AoE ability in a 100-player event would generate 100 damage resolution events per activation, consuming a disproportionate share of the server's tick budget.
  • VFX tiers per ability: Every ability is designed with 3 VFX tiers: full (under 30 participants in the local area), reduced (30 to 60 participants), and minimal (60+ participants). The combat designer specifies these tiers as part of the ability design, not as a post-hoc artist optimisation. GW2 WvW ships this pattern.
  • Death respawn at safe locations: When a player dies, they respawn at the nearest safe waypoint, not at the combat site. This prevents the "death ball" pattern where players die and respawn in the same overcrowded combat area, creating an ever-growing density problem.
  • PvP requires opt-in flag: In the open world, soft-lock on hostile players requires the attacker to be PvP-flagged. The flag state is a single replicated boolean, which is negligible in bandwidth. This prevents accidental PvP and keeps the networking model simple (unflagged players are not valid combat targets and do not need combat-priority networking).
  • Combat lockout at shard boundaries: Players in active combat (PvP flagged with a target lock, or aggro'd by NPCs) cannot enter shard transition zones (the gates between overworld and Downtime). This prevents the exploit of fleeing into a shard boundary for invulnerability during the transition period.

PvE Instance Networking

Dungeons, raids, and scripted encounters run on separate dedicated server processes at 30 Hz tick rate. These are spawned on demand by Agones from a warm pool of pre-initialised server processes for near-instant availability. Player counts are small and predictable (4-player dungeons, 8-player raids, 24-player alliance raids), so these servers do not need interest management, staggered replication, or the other complexity of the overworld zone server. They are standard UE5 dedicated server instances with default replication, which is entirely sufficient at these player counts.

Structured PvP Networking

Arenas (3v3, 5v5) and battlegrounds (15v15, 20v20) run on separate dedicated server instances at 30 Hz. Siege and territory war (50v50 target) runs on dedicated instances at 20 Hz target. These are prototype targets, not commitments, until validated under synthetic and real playtest load.

7. Security and Resilience

7.1 DDoS Mitigation

The Problem

An MMO with publicly addressable game servers is a DDoS target. If a game server's IP address is known to clients, an attacker can flood that IP with traffic and take down a zone server with 200 to 300 players connected. The server IP is typically embedded in the connection address sent to the client during matchmaking or zone assignment.

The Solution

Cloudflare Spectrum Enterprise proxies all game traffic (UDP and TCP) through Cloudflare's edge network. The client connects to a Cloudflare edge IP, not the game server's real IP. Cloudflare forwards legitimate traffic to the origin server. DDoS traffic is absorbed at the edge. This is the only way to hide the game server's IP from clients in the recommended Agones/EKS architecture.

AWS Shield Standard provides baseline volumetric DDoS protection for AWS resources but does NOT hide origin IPs. It mitigates attacks on known IPs; it does not prevent attackers from discovering those IPs. Shield Standard is included at no cost with AWS and should be active, but it is not sufficient on its own.

AWS GameLift DDoS Protection (launched March 2026) provides IP hiding via the GameLift Player Gateway, but only for games running on GameLift managed or container fleets. This is not compatible with the recommended Agones/EKS architecture unless the studio switches to GameLift for hosting.

Critical UDP caveat: Cloudflare Spectrum drops fragmented UDP packets at the edge. The game's network protocol must keep UDP datagrams below path MTU (typically 1,400 bytes for internet traffic). UE5's default packet sizes are generally within this limit, but custom RPCs or large batch replication events must be checked.

7.2 Anti-Cheat

Layer 1: Server Authority

The server validates ALL state-changing inputs. Movement speed, ability damage, item acquisition, and combat results are computed server-side. The client sends inputs (move direction, ability activation, target selection); the server processes them and sends authoritative results back. A cheating client cannot give itself items, teleport, deal extra damage, or ignore cooldowns because the server does not trust client state for any of these.

Interest management provides a secondary anti-cheat benefit: by only sending clients entities they could plausibly observe (based on distance and the Replication Graph's spatial filtering), the server reduces the amount of game state available to hack. However, interest management does not replace anti-cheat. Actors within the relevancy radius are still fully visible to the client, and a modified client could extract information from those actors that a normal client would not display.

Layer 2: EasyAntiCheat

Base EAC is free via Epic Online Services and has native UE5 integration. It provides client-side anti-cheat with integrity checks and telemetry that helps detect common aimbots, speed hacks (clients sending plausible-looking but inhuman input sequences), tampering, and memory manipulation. EAC supports dedicated server topologies natively. Advanced support or enterprise features may require a separate agreement with Epic.

7.3 Server Crash Recovery

Durable State vs Hot State

PostgreSQL is the durable authority for all important player state. Player data is written to PostgreSQL on every significant state change: level up, item acquisition, quest completion, trade confirmation, and disconnect. Writes must be idempotent, queued/backpressured where appropriate, and load-tested against peak event cadence. If a server crashes and Redis is lost, PostgreSQL has a complete record of everything that matters.

Redis holds hot transient state for fast recovery: current position, health, mana, combat state, active buffs. This state is snapshotted to Redis every 30 to 60 seconds. If a server crashes, the transient state from the last Redis snapshot can be loaded to resume players near where they were. At worst, players lose 30 to 60 seconds of positional progress. Redis is a cache, not the recovery authority.

Crash Recovery Flow

  1. Zone server process crashes. Agones detects the health check failure (5 to 10 second interval).
  2. All connected players are disconnected. Clients display a "Reconnecting..." screen.
  3. Agones allocates a replacement zone server from the warm pool (1 to 5 seconds).
  4. The replacement server loads the zone level (10 to 30 seconds depending on zone complexity and whether the warm pool server has the level pre-loaded).
  5. The replacement server loads durable player state from PostgreSQL and recent transient state from Redis.
  6. Players reconnect via the gateway service and resume from their recovered state.

Target recovery time: 60 to 120 seconds. The cold zone level load is the bottleneck. If warm pool servers have the level pre-loaded, recovery time drops to 30 to 60 seconds. Hot standby is excluded from baseline because it materially increases infrastructure cost and operational complexity.

7.4 Console Compatibility

The architecture is PC-first with console to follow. Dedicated servers avoid peer-host NAT traversal (clients make outbound connections to the server, not to each other), but PSN/XBL session integration, invite handling, reconnect behaviour, and TRC/XR certification requirements must be validated separately. These are integration concerns, not architecture changes. Build platform-specific session and reconnection requirements into the design from the start to avoid costly retrofitting.

10. What We Do Not Know

Significant unknowns that must be resolved by prototyping. Presenting them honestly is more valuable than pretending certainty.

  1. The actual single-server player ceiling for this game. No published data exists for a third-person MMORPG with hybrid combat and cosmetic customisation at 300+ players in UE5. The 200-player prototype resolves this.
  2. Replication Graph performance with CH's specific actor complexity. The no-stat cosmetic system means player characters carry more visual state than a typical MMO character (full wardrobe, dye state, weapon crystals, off-hand visuals). Each additional replicated property affects bandwidth. Must be measured, not assumed.
  3. Actual ClientTravel transition time with full cosmetic state. Serialising and deserialising a character with extensive cosmetic customisation through Redis may take longer than a standard combat character. Must be prototyped.
  4. MassAI client-side deterministic simulation viability. Client-side NPC simulation (server sends seed state, client simulates locally) only works if all clients produce identical results from the same inputs. Floating-point determinism across diverse hardware is not guaranteed. Must be tested across at least 3 hardware configurations.
  5. If Replication Graph proves insufficient. No lower-risk path has been identified for CH's target without reducing population, changing encounter design, or adopting licensed meshing (OmniMesh).
  6. OmniMesh licensing terms. Pricing, source code access, support, and platform restrictions are not publicly documented.
  7. Min-spec client rendering at 50+ nearby characters with CH's art style. The Byte-Punk aesthetic may be more or less expensive to render than a generic MMO character. Must be tested on target minimum-spec hardware.
  8. Custom metagame backend scalability at 30,000+ CCU. Must be designed for horizontal scaling from the start. Load testing required before launch.
