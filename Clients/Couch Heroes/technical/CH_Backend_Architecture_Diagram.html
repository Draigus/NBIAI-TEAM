Orchestration and Backend Services
Core
Agones (Kubernetes)
Open-source game server lifecycle management
Allocates, scales, health-checks all game servers
Warm pool management for instant instance availability
Cloud-agnostic (runs on AWS EKS, GCP GKE, or bare metal)
No vendor lock-in
Custom
Custom Metagame Backend
Purpose-built for Couch Heroes. No third-party managed service.
Core services: Accounts and authentication, social graph (friends, guild, faction membership), matchmaking, shard routing (social-graph-aware assignment), economy and trading (centralised marketplace), faction state management
Deferred to post-launch: Advanced chat features, economy automation, faction analytics, live ops tooling
Horizontally scalable on Kubernetes
Hardware: 3x c7i.xlarge cluster
Corruption World-State Service
Single source of truth for corruption data
Per-zone scalar (0 to 100) + 64x64 spatial grid
Versioned snapshots for late-join full-state recovery
Idempotent cell updates
Pub/sub to all zone servers and hub shards (1 to 5s eventual consistency)
Backed by PostgreSQL
Shard Router
Part of the metagame backend
Routes players to Downtime shards based on social graph
Manages shard creation and destruction
Handles cross-shard party invites and transfers
Telemetry Service
Per-zone player counts
Per-connection bandwidth
Server tick time and NetBroadcastTickTime
Combat event participation
Shard transition success/failure rates
Dashboarded (Grafana)
Data Layer
Durable Authority
PostgreSQL (RDS Multi-AZ)
The single source of truth for all persistent state.
Player data: level, inventory, quest progress, cosmetic collection, skill allocations
Economy: currency balances, marketplace listings, trade history
World state: corruption scalars and grids, faction territory control
Faction data: membership, reputation, rankings
Writes on significant state changes (idempotent, queued, backpressured, load-tested against peak event cadence)
If Redis is lost, PostgreSQL has everything that matters
Hardware: db.r6g.2xlarge Multi-AZ
Hot Cache
Redis (ElastiCache Cluster)
Ephemeral state transfer and hot cache. Not the recovery authority.
Player state during server transitions (120s TTL)
Position, health, combat state snapshots (30 to 60 second intervals)
Session tokens and matchmaking state
Verification handshake: receiving server confirms load before source releases connection
On failure: player returned to source server, not dropped
Hardware: 3-node cache.r6g.xlarge cluster
Data Flows
→ Players connect through Cloudflare Spectrum to game servers UDP proxied
↔ Zone servers read/write player state to PostgreSQL on significant changes
↔ Zone servers snapshot hot state to Redis every 30 to 60 seconds
← Zone servers subscribe to Corruption service pub/sub, 1 to 5s latency
→ Shard Router assigns players to hub shards via social graph on zone entry
↔ Server transitions serialise player state through Redis ClientTravel + 120s TTL
→ Agones manages all game server lifecycle allocate, scale, health-check
→ Metagame backend handles auth, matchmaking, economy all non-simulation state
→ Telemetry collects from all servers per-connection metrics
→ EasyAntiCheat runs client-side, reports to server authority complementary layers
Player Journey: Key Server Transitions

Login to Zone

1
Client authenticates via metagame backend
2
Matchmaking assigns zone based on friends, faction, population
3
Client receives Cloudflare Spectrum endpoint (not raw server IP)
4
UDP connection established through Spectrum to zone server
5
Zone server loads player state from PostgreSQL
6
Player spawns in zone, begins receiving interest-managed updates

Zone to Downtime Hub

1
Player enters shard transition zone (city gate)
2
Combat lockout check (cannot transition while in combat)
3
Shard Router queries social graph, assigns shard
4
Zone server serialises player state to Redis (120s TTL)
5
ClientTravel() to hub shard (3 to 5 second transition, masked by gate animation)
6
Hub shard loads state from Redis, confirms via handshake
7
Source server releases connection only after confirmation

Entering a Dungeon

1
Party leader initiates dungeon entry
2
Metagame backend requests instance from Agones warm pool
3
Agones allocates a pre-initialised dungeon server (near-instant)
4
All party members serialise state to Redis
5
ClientTravel() to dungeon instance (30 Hz, 4-player)
6
On completion/exit: reverse flow back to zone server
7
Empty dungeon instance returned to pool by Agones

Visiting a User Space

1
Player interacts with House of Houses portal in Downtime
2
Backend checks target User Space: is housing server loaded?
3
If not loaded: housing server loads sub-level via World Partition
4
Player state serialised to Redis
5
ClientTravel() to housing server (portal animation mask)
6
Player arrives in User Space with up to 24 visitors

Server Crash Recovery

1
Zone server crashes. Agones detects via health check (5 to 10 seconds)
2
All players disconnected. Clients show "Reconnecting..."
3
Agones allocates replacement from warm pool (1 to 5 seconds)
4
Replacement loads zone level (10 to 30 seconds)
5
Loads durable state from PostgreSQL + hot state from Redis
6
Players reconnect via gateway. Resume from recovered state.
7
Total recovery: 60 to 120 seconds

Corruption Event Flow

1
Corruption event triggers on zone server
2
Participation zone defined. First 100 players become active.
3
Active participants receive NetPriority 3.0 elevation
4
Additional arrivals become observers (reduced LOD, no interaction)
5
On cleansing: Corruption Manager wakes, replicates grid delta
6
Change propagates to Corruption World-State Service
7
Service fans out to all hub shards (1 to 5 second latency)
