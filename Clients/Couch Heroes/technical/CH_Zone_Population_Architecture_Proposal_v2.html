0. Decision Framework

This document asks Couch Heroes leadership to make one architectural commitment: how many concurrent players per zone should the networking architecture support?

The answer is not a single number. It is a spectrum with different engineering costs, different team requirements, and different competitive positioning.

Recommendation

Build the architecture for 200–300 players per zone. Prototype at 200 first. Let telemetry from the prototype determine the achievable ceiling at acceptable quality.

Critical Constraint — All Readers Must Understand

Regardless of zone population, any single combat encounter (world boss, corruption event, PvP battle) caps at approximately 100 participants. Players beyond that cap can observe but not interact. The zone holds 200–300 players coexisting; combat happens in clusters of up to 100 within that zone. This is the standard MMO pattern — the zone population and the combat population are different numbers.

1. Executive Summary

Couch Heroes is a fantasy Social MMORPG built in Unreal Engine 5, targeting PC first with console to follow. The game's design pillars — discovery-driven exploration, a corruption world mechanic, no-stat cosmetics, soft-class building via weapon + off-hand, and a social-first identity — demand a world that feels alive and populated.

The original GDD specified 30–40 players per instance. This proposal defines the architecture required to scale zone populations to 200–300 concurrent players per zone, with hub areas (Downtime, faction districts) scaling dynamically through sharding, and combat encounters running responsively at up to 100 participants.

Why population density matters for Couch Heroes specifically

Couch Heroes is a Social MMO. The social experience IS the product. A world with 30 players per zone feels empty — Palia is a strong cautionary example: its low visible population cap was widely criticised by players, and its commercial outcome shows the risk of under-populated social MMO spaces. Couch Heroes' five factions (Allegiant, Athenaeum, Adeticus, Mendarium, Purple) only feel meaningful when faction districts are populated. The corruption mechanic only generates shared stories when multiple players witness and participate in corruption events. The no-stat cosmetic system only drives aspiration when players see each other's outfits.

Social MMO competitors with gameplay systems cap at 25–50 players per zone (Palia, Tower of Fantasy, Destiny 2). Social virtual worlds without combat reach higher (VRChat at 80, Second Life at 100) but lack game systems. FFXIV achieves ~300 per zone with all combat instanced on separate servers and a custom engine built over 15 years. A UE5 Social MMO that delivers 200–300 players coexisting in a zone with real-time open-world combat would be a genuine differentiator: approaching FFXIV's zone density while supporting open-world PvE and PvP on the zone server.

What worked

  • FFXIV's city sharding — cleanest model for Downtime. Friends/party routed to same instance. Players can switch manually.
  • GW2's megaserver — dynamic zone instancing with friend/guild-aware routing creates perception of a populated world.
  • GW2's centralised Trading Post — solves cross-shard commerce. Global service outside the instance system.
  • PlanetSide 2 — hundreds of players in active combat via aggressive distance-based update rates.

What failed

Palia — The "Empty World" Problem

25-player cap in a social game. World felt empty. Community cited loneliness as a core issue. Two rounds of layoffs, then acquired. Lesson: a Social MMO must feel populated or it fails at its core promise.

Throne and Liberty — The "Marketing vs Reality" Problem

Marketed large-scale sieges. Community reported freezing, severe input lag, visual chaos. 107 → 25 servers in 4 months. Lesson: claiming large-scale and delivering smooth large-scale are different things.

ESO Cyrodiil — The "Retrofit" Problem

Decade of large-scale PvP performance issues. Now testing simplified abilities to reduce server load. Lesson: combat must be designed networking-friendly from the start, not retrofitted.

Tower of Fantasy — The "Silent Split" Problem

Auto-channelling silently separates players ignoring social connections. World never feels populated. Lesson: channel splitting that ignores social connections destroys the social experience.

Where players are at any given moment

300 players in a zone does not mean 300 in one place. A plausible busy-state distribution:

  • ~40–80 in Downtime areas (handled by hub shards separately)
  • ~30–80 in active combat clusters (corruption events, world bosses, PvP)
  • ~120–200 exploring, questing, gathering, fishing — spread across the zone
  • ~20–50 transitioning to or from instanced content (dungeons, arenas, User Spaces)

These are illustrative proportions, not hard allocations. The actual mix will vary by time of day, content cadence, and event scheduling.

Corruption State Replication

Corruption is a core design pillar. The networking architecture supports it natively with a tiered hybrid model:

  • Per-zone corruption scalar (0–100, one byte) — the authoritative gameplay value driving spawns, accessibility gates, and VFX palette. Replicated to all clients in periodic world-state snapshots. Essentially free in bandwidth terms.
  • Low-resolution spatial grid (64×64 cells, 1 byte per cell = 4 KB total) — visual corruption spread patterns. Replicated via delta compression (only changed cells sent). Typical update: 50–200 bytes every 5–10 seconds. A corruption update costs usually smaller than one full moving character update under this replication model.

Cross-shard consistency: corruption state owned by a central world-state service (PostgreSQL + pub/sub). Changes fan out to all zone instances and hub shards with 1–5 second latency. Acceptable for environmental state.

User Space (Housing) Architecture

User Spaces are personal pocket dimensions accessed via the House of Houses. Shared housing server processes host 20–50 concurrent occupied User Spaces via UE5 World Partition streaming. Visitor cap: 12–24 per house (consistent with ESO and FFXIV). Entry is a server transition via ClientTravel() masked by a portal animation.

5. Technical Specification

5.1 Overworld Zone Server

  • Single UE5 dedicated server process per zone
  • Target: 200–300 concurrent connections.
  • Server tick rate: 10–15Hz realistic at 300+. 20Hz achievable at 200 or fewer.
  • Staggered replication: 50–100 connections per tick, rotated. Priority accumulation ensures combat participants are never starved.

Interest Management Tiers (with Hysteresis)

*Effective Rates are Lower than Tick Rate Due to Staggered Replication

At 300 connections with 50–100 replicated per tick at 15Hz, each connection receives a replication pass every 3–6 ticks (200–400ms). Near-tier players with elevated priority (3.0) win passes more frequently but still cannot achieve the full tick rate. The effective update rate for near-tier at 300 players is ~3–10Hz, not 15Hz. At 200 players, near-tier approaches 10–15Hz. This is why the 200-player prototype is essential — it determines whether these effective rates deliver acceptable combat responsiveness.

Hysteresis Prevents Tier Thrashing (Custom Code Required)

Hysteresis (separate promote/demote distances) is not a stock UE5 feature — it must be implemented in the custom Replication Graph node. Without it, players near tier boundaries oscillate between update rates causing visual popping. The 10m buffer (promote at 45m, demote at 55m) is the minimum guard.

5.4 The 100-Participant Event Cap

Open-world events (corruption cleansing, world bosses) cap at 100 active participants:

  • At 100 participants, event transitions to "full" state
  • Additional players can observe (reduced-LOD visuals) but cannot deal damage or receive loot
  • When a participant leaves, a slot opens for an observer
  • Robin's team should frame this as "the corruption is overwhelming — reinforcements standing by"

5.5 Combat Design for Networking

Soft-Lock Skill Expression

Positioning-Based

  • Positional bonuses — flanking/rear attacks deal more damage (FFXIV)
  • Dodge i-frames — invincibility frames with precise timing (GW2)
  • Momentum dodges — movement speed/direction affects dodge trajectory

Lock Management

  • Lock degradation — lock weakens when target dashes or enters cover
  • Split-body animation — attack locked, lower body free to move (Ashes of Creation)
  • Animation cancelling — cancel recovery into startup of next ability

Networking-Friendly Design Rules

  • AoE caps at 12 targets (nearest by distance) — prevents O(N²) scaling
  • VFX tiers: full (under 30 players), reduced (30–60), minimal (60+) — defined by combat design, not post-hoc optimisation
  • Death respawn at safe locations — prevents "death ball" density buildup
  • PvP requires opt-in flag — single replicated boolean, negligible bandwidth

6. Security and Resilience

DDoS Mitigation

AWS Shield Standard does NOT hide server IPs — it only mitigates volumetric attacks on known IPs. To actually hide game server IPs from clients, game traffic must be proxied through Cloudflare Spectrum (Enterprise plan, UDP proxy) or GameLift Player Gateway (only available with GameLift managed/container fleets, not with Agones/EKS). With the primary Agones/EKS architecture, Cloudflare Spectrum is required for IP hiding. Without it, server IPs are exposed to clients via the connection address.

Anti-Cheat

Two layers: server authority (validates all state-changing inputs; interest management reduces remote-state exposure for irrelevant actors but does not replace visibility checks for relevant actors) + EasyAntiCheat (base free via EOS; catches aimbots and memory manipulation; enterprise support may require separate agreement).

Crash Recovery

PostgreSQL is the durable authority for player state — written on every significant state change (level up, item acquisition, quest completion, disconnect). Redis holds hot transient state (position, health, combat state) with 30–60 second snapshots for fast recovery. On crash: Agones detects failure, allocates warm-pool replacement, loads durable state from PostgreSQL + recent snapshot from Redis. Players see "Reconnecting..." screen. Target recovery: 60–120 seconds (cold zone load is the bottleneck). Redis is a cache, not recovery truth.

Console Compatibility

PC-first architecture is console-compatible. Dedicated servers avoid peer-host NAT traversal (clients make outbound connections), but PSN/XBL session integration and TRC/XR reconnect behaviour must be validated separately. Build platform-specific session and reconnection requirements in from the start.

7. Expert Perspectives

Network Engineer

Custom Engineering Clarity

UE5 provides the Replication Graph plugin with GridSpatialization2D for spatial culling, NetUpdateFrequency / NetPriority for per-actor tuning, and dormancy states. Everything else in this architecture is custom C++ that must be built: distance-based frequency tiers (custom Replication Graph node), hysteresis, combat priority elevation logic, staggered replication budgeting, impostor billboard rendering for skeletal meshes, and client-side NPC simulation. The Replication Graph gives you the framework; the game-specific interest management is engineering work.

  • Replication Graph is the foundation. Iris is Experimental and should not be planned around. If the Replication Graph ceiling is too low, the answer is OmniMesh or reduced targets, not an experimental API.
  • Prototype at 200 players immediately. Measure NetBroadcastTickTime, per-connection bandwidth, server tick time. This prototype is the gate for all further work.
  • Reliable RPC discipline is a code review gate. No reliable RPCs in combat paths. A single reliable RPC in a hot path stalls replication for an entire connection.
  • Interest management hysteresis is mandatory. 10m buffer minimum between promote and demote distances. This is custom code, not a stock UE5 feature.
Engineering Estimate

Networking layer (interest management, dormancy, priority accumulation, staggered replication, state transfer, hub sharding) requires 2–3 senior engineers dedicated to infrastructure for pre-production and early production.

Level Designer

  • Initial zone size target: 2km × 2km to distribute 300 players naturally (validate during Phase 1 — this is a design heuristic, not a proven constraint)
  • No single POI attracts more than 100 players
  • Downtime's five faction districts distribute key NPCs across districts
  • Natural density barriers (terrain, verticality) prevent convergence
  • Shard transition zones use natural geography (city gates, bridges, tunnels) to mask 3–5 second transitions

Combat Designer

  • AoE target caps from day one — every AoE caps at 12
  • VFX tiers in the ability pipeline — 3 tiers per ability, specified by combat design
  • Corruption cleansing as positional puzzle/platforming — inherently networking-cheap
  • Death respawn at safe locations away from combat sites

Product Manager

  • Lead with experience: "You always see other players" — not raw player counts
  • The 100-player event cap is a feature: "epic battles with up to 100 combatants"
  • MassAI NPC crowds make 100 human players feel like a living world
  • New player test at every milestone: what does the first 30 minutes look like at 100 concurrent vs 300?

CTO / Full Stack

  • Containerise everything on Kubernetes. Agones manages game server lifecycle.
  • Telemetry from day one — dashboard per-zone, per-connection, per-event
  • Blue-green deployment for stateless backend services (metagame backend, world state service). For zone servers: scheduled draining — stop accepting new players, wait for natural zone transitions to empty the server, then replace. This is not instant and requires planned maintenance windows for major server updates.
  • State transfer with verification handshake. Return to source on failure, not dropped.
Backend Engineering Estimate

Infrastructure (Kubernetes, Agones, custom metagame backend, Redis/PostgreSQL, telemetry, CI/CD, DDoS) requires 2–3 backend/DevOps engineers during pre-production, then ongoing ops support.

10. What We Don't Know

Significant unknowns that must be resolved by prototyping. Presenting them honestly is more valuable than pretending certainty.

  1. The actual single-server player ceiling for this game. No published data for a third-person MMORPG with hybrid combat and cosmetic customisation at 300+ in UE5. The stress test resolves this.
  2. Replication Graph performance with CH's actor complexity. The no-stat cosmetic system means more visual state per character than a typical MMO. Must be measured.
  3. Actual ClientTravel transition time with full cosmetic state through Redis. Must be prototyped.
  4. MassAI client-side deterministic simulation viability. Floating-point determinism across diverse hardware is not guaranteed.
  5. If Replication Graph proves insufficient. Iris is Experimental and cannot be relied on. The fallback is OmniMesh licensing or reduced zone population targets.
  6. OmniMesh licensing terms. Pricing and source code access not publicly documented.
  7. Min-spec client rendering at 50+ characters with CH's Byte-Punk art style. Must be tested.
  8. Custom metagame backend scalability at 30,000+ CCU. Must be designed for horizontal scaling from the start. Load testing required.

11. Phased Delivery

Phase 1: Foundation (Pre-production)
Validate the architecture with hard data
  • 200-player stress test prototype with Replication Graph
  • Shard transition prototype (ClientTravel + Redis round-trip)
  • MassAI determinism test across 3+ hardware configs
  • OmniMesh licensing inquiry (only if prototype ceiling is below the required target range, or if 300 remains a hard business requirement after Phase 1)
  • Custom metagame backend prototype (accounts, social graph, matchmaking) on Kubernetes

Team: 2–3 networking engineers + 1 DevOps + 1 backend engineer, full-time

Gate

Prototype must pass ALL of these before proceeding:

  • Server tick rate > 15Hz sustained (not average — p99 must not drop below 10Hz)
  • NetBroadcastTickTime < 40% of tick budget
  • Per-connection bandwidth < 50 KB/s at steady state
  • No server-induced packet loss; measured loss remains within controlled test-network baseline
  • Reliable RPC queue depth stays bounded (no unbounded growth)
  • Memory and GC stable over 1-hour run (no leaks)
  • 100-player combat cluster test: combat responsiveness subjectively acceptable at 100ms RTT
  • Client rendering at 50+ nearby characters on minimum-spec hardware: > 30 FPS

If any of these fail → reduce target and/or evaluate OmniMesh.

Phase 2: Core Infrastructure (Early Production)
Build the production networking layer
  • Interest management with three tiers, hysteresis, combat elevation
  • Hub sharding for Downtime with social-graph routing
  • State transfer pipeline (Redis + verification handshake)
  • Instance server pipeline via Agones
  • User Space housing server pipeline
  • Corruption state replication
  • Telemetry dashboard + EasyAntiCheat + DDoS proxy

Team: 2–3 networking + 1–2 DevOps + 1–2 backend for metagame/world state

Phase 3: Scale and Polish (Mid-Late Production)
Optimise, stress test at full target, prepare for launch
  • 300-player stress tests under production conditions
  • Continuous stress testing in CI
  • Server crash recovery validation
  • Console compatibility (PSN/XBL session integration)
  • Min-spec client performance profiling
  • All thresholds data-driven and tunable via config
